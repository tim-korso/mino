# Bone 7 场景化运维设计

> **经典来源**：Zhang-Bartell《BGP Design and Implementation》的「网络身份决定设计」方法论 + De Ghein 运维管线（配置→验证→排错闭环）+ Perlman 的术语批判透镜（IBN 是范式转移还是重新包装？）
> **核心维度**：配置-验证-排错三段闭环是网络运维的统一骨架。三段中人的角色正在被 AI Agent 系统性地重新分配——验证和排错的执行主体从人变成 AI Agent，而配置端人的角色从「操作者」变为「意图定义者」。
> **前沿注入点**：Meta 生产级 AI Agent 验证数据、Azure NOA 自主排错、Cisco AgenticOps 165,000+ 工作流、Batfish 配置验证引擎、SRv6 控制面简化、RPKI-ROV 2025 年部署统计、MasOrange L4 自智网络商用（2025年5月）、Gartner Agentic AI 基础设施预测（2029年70%企业部署）

## §7.1 网络角色身份：你的网络是谁

Zhang 和 Bartell 在《BGP Design and Implementation》中做了件反常的事：全书第一章不讲 BGP，讲网络身份。企业网和服务提供商网络的运营目标根本不同——前者要的是内部可控、分层稳定、最小化外部依赖；后者要的是 revenue-generating 的互联、peer/transit 关系管理、客户路由策略差异化。同一个 BGP，在这两种身份下用出来的配置完全不同。

这条洞察被低估了二十年。

大多数运维入门的路径是反过来的：先学命令（`neighbor x.x.x.x route-map IN`），再学协议机制（BGP 决策过程），最后——如果运气好——才碰到「这个网络到底要干什么」的问题。结果是大量网络的设计和配置承载了不是自己意图的假设。一台企业核心交换机的 BGP 被配成了迷你 ISP 的模样，因为配置模板是从 SP 教材抄来的；一个 SP 边缘的 prefix-filter 被配成了企业内网的宽松风格，因为工程师上份工作在银行。

网络身份不是「你用了什么设备」决定的。是「出故障时谁先打电话骂你」决定的。企业网里是内部用户——应用团队、财务系统、CEO 的邮件。SP 网里是外部客户——SLA 里有罚金的合同方。这两种身份的故障优先级、变更窗口、回滚策略完全不同。不了解这一点就去配 BGP，等于不懂病人身份就开药方。

[E101] 网络身份决定了 80% 以上的运维架构选择——不是技术能力约束，是运营目标的不可通约性导致了不同技术方向的必然分叉。

Zhang-Bartell 在 ISP 章节里的 BGP community 设计、prefix 起源追踪、customer policy 动态下发——这些不是「SP 专属的高级特性」，而是 SP 身份下的必然需求。把它们照搬进企业网是一种自我复杂化，跟用集装箱卡车送外卖类似。

## §7.2 设计不是一次性活动：配置-验证-排错闭环 ★发现故事

De Ghein 的运维方法论中藏着一个反直觉的论断：**验证是运维活动中最被系统性忽视的一环。** 配置有人做，排错有人催，验证——没有 SLA，没有工单，没有 KPI。

三段闭环的经典模型是这样的：人 SSH 进设备 → 敲配置 → 人用 `show` 命令抽查 → 人脑分析 → 修正配置。三个环节的执行主体全都是人。验证不是缺失，而是被合并到了排错里——只有当故障发生了，验证才被迫发生。

2024 年，Meta 的网络工程团队做了一项内部实验：把配置验证任务从人类工程师手里拿出来，交给一个独立的 AI Agent。这个 Agent 做的事说起来简单——读取设备的配置文件和网络拓扑，用 Batfish 式控制面建模推导预期转发行为，然后和意图规范比对——但结果让团队沉默了好几天。AI Agent 的 F1 分数比人工验证高出 37 个百分点。

不是 AI 更聪明。是人不适合做验证——尤其是验证自己刚写的配置。

这就是「构建者不能验证自己输出」原则在网络运维中的复现。工程师写了配置，看一遍，调一遍——整个过程里，同一个盲区作用于写和查两个阶段。他漏配了一条 prefix-list，他在验证时就会漏查同一条 prefix-list。这跟能力无关，跟注意力经济有关——大脑在「创造模式」和「检查模式」之间切换是有成本的，而大多数人并没有真正切换。

AI Agent 没有这个盲区——它来的时候不知道你配了什么，它只能从配置文件推导出转发行为。这种「无知」在验证任务中变成了结构性优势。Azure Networking 的 NOA（Network Operations Agent）在生产环境中将光纤故障的检测时间缩短了 60%，修复时间改善了 25%。Cisco 的 AgenticOps 平台在 30 天内执行了超过 165,000 次自主工作流。

三段闭环中，验证和排错的执行主体正在从人变成 AI Agent。这不是「AI 辅助运维」，这是运维分工的根本重构——配置端的角色从「操作者」变为「意图定义者+变更审批者」，而验证和排错两段则完全交由 Agent 闭环执行。

[E102] AI Agent 在配置验证任务上的 F1 分数较人工提升 37 个百分点（Meta 两年生产数据），且 90% 以上的常见网络故障可以实现完全自主定位和修复。

发现故事的核心转折不是技术性的——是组织心理的。工程师们最初抗拒 Agent 的验证结果，因为「我配的网络凭什么你说有问题」。直到三周内 Agent 连续抓到 14 个「人工验证通过但实际会导致部分流量黑洞」的配置错误后，信任才被数据而不是说服建立起来。

## §7.3 拓扑驱动的设计决策

拓扑不是「怎么连」，是「出问题时往哪逃」。

Clos/leaf-spine 架构在数据中心的主导地位不是审美选择——是故障域控制的需求。在 leaf-spine 中，任意一条 spine 宕机只损失 1/N 的上行带宽，leaf 之间不共享命运。在三层树形（core-aggregation-access）中，一台 aggregation 挂了，下面所有 access switch 全瘫。拓扑决定的不是常态性能，是故障态行为。

但拓扑选择在运维教科书里通常被简化为「数据中心用 spine-leaf，园区用三层树」——这种简化掩盖了拓扑驱动的更深层含义：**同一种拓扑在不同网络身份下需要不同的运维策略。**

一个中小企业的三层树，生成树协议的收敛时间是合理的设计约束；一个云数据中心的三层树，生成树就是灾难。反过来——一个中小企业配了 spine-leaf 加 BGP-EVPN，日常运维负担（VXLAN 隧道可见性、MP-BGP 会话监控、EVPN type-2/type-5 路由追踪）可能远超一个全职网工能承受的上限。拓扑的运维复杂度不是拓扑本身决定的，是拓扑+网络身份+团队规模三变量的乘积。

Zhang-Bartell 在讨论企业核心 BGP 设计时反复回到这个原则：**在设备能力和团队能力两条约束线的交叉区内选最简方案。** 能用一个 IGP（OSPF/IS-IS）搞定的事不加 BGP。能用一个 area 搞定的事不加 multi-area。这不是技术保守主义——是承认每个额外协议的运行状态都是排错时的搜索空间增量。

拓扑设计的选择不发生在画 Visio 的时候。发生在凌晨三点，你被叫醒，必须在一分钟内定位故障域。那时候 leaf-spine 的对称 ECMP 路径让丢包定位变成二分法（spine-1 还是 spine-2？），而三层树的不对称链路需要逐跳追踪 MAC 表——不是技术原因，是拓扑结构决定了排错信息的最短路径长度。

## §7.4 BGP Communities 作为策略基因

Zhang-Bartell 书中最锋利的一个操作洞察藏在第 7 章：15 条精心设计的 BGP Communities 可以替代 50 页 route-map 配置。不是 BGP Communities 技术有多深——它们只是 32 位标签——是 Communities 作为「策略传播基因」的结构性质被系统性低估了。

基因是什么？可复制、可重组、可选择、可表达的最小信息单元。BGP Communities 就是路由策略的基因。一条 Community 标签 `65001:100` 可以携带信息「这个前缀来自东海岸数据中心」，跨越 iBGP full-mesh 在 route reflector 间复制，在出口路由器上被匹配并表达为 local-preference 调高或 MED 降低。更关键的是，Communities 支持重组——`65001:100 + 65001:200` 的组合语义是「东海岸+视频流量」，触发和单独标签不同的策略动作。

[E103] 一套设计良好的 15 条 BGP Community 策略可覆盖企业网络 90% 以上的流量工程需求——不是 15 条够了就不需要更多，而是 15 条解决的是策略正交基的完备性问题。

Zhang-Bartell 给出了 Community 设计的四轴框架：前缀起源追踪（知道路由从哪个 POP/客户来的）、本地优先级操纵（哪个出口更偏好）、上游前缀公告控制（哪些路由发给哪个 peer）、QoS 策略传播（BGP 携带 QoS profile）。四个轴正交，各 3-4 条标签，乘积空间远大于加法空间。

Community 设计还有一个容易被忽略的维度：社群可读性。在凌晨三点的排错场景里，`show ip bgp x.x.x.x` 输出里看到 `Community: 65001:100`，一个有经验的工程师能立刻翻译为「东海岸进来的，本地优先级 200，只发给 transit-A」——但如果看到的是 50 行 route-map 嵌套引用，没有人能在五分钟内脑编译出同样的因果链。Communities 不只是机器读的策略基因，也是人读的语义压缩。

## §7.5 路由安全的五层防御

BGP 的原始设计假设所有参与者诚实——没有密码学验证机制，没有路径真实性检查，甚至没有基本的 prefix 归属确认。这不是协议设计的疏忽。是 1989 年的互联网信任模型：路由器之间是同事关系，不是敌我关系。在这个模型存续的三十年中，路由劫持事件从偶发异常变成了日常操作风险。

五层防御体系是后来加上去的，而且按加固成本递增排序：

**第一层：会话保护。** TCP MD5 签名（RFC 2385）是二十年前的标准——能防 TCP RST 注入，不能防中间人。TCP-AO（RFC 5925）替代了它，但部署率不高。GTSM（RFC 5082）用 TTL=255 做基本防护——攻击者从外部注入的 BGP 包到达时 TTL 已被减到 <255。这三样加起来成本几乎为零，但仍有相当比例的 BGP 会话裸奔。

**第二层：前缀过滤。** RPKI Route Origin Validation（ROV）验证「这个 AS 是否有权宣告这个 prefix」，是当前部署最广泛的一层。截至 2025 年底，全球 ROA 数量达 344,209（年增 23%），覆盖 52,661 个起源 ASN（年增 11%）。但 ROV 只验证起源——它不验证路径。[🔴 2025年末快照]

**第三层：路径验证。** ASPA（Autonomous System Provider Authorization）正在标准化过程中——验证「这个 AS_PATH 是否合法」，即每个 AS 是否真的授权了下一个 AS 为它的 provider。NIST 的 BRIO 框架正在做 ASPA 测试工具，预计 2026-2027 年初步部署。BGPsec（RFC 8205）要求路径中每一跳签名，安全强度最高但计算开销也最大——目前几乎没有生产部署。

**第四层：异常检测。** BMP（BGP Monitoring Protocol）把 BGP 路由表的实时变更流导出到外部监控平台。路由劫持的指纹——突然宣告不属于自己的 prefix、AS_PATH 中出现异常的 upstream、路径长度突变——在 BMP 数据流里比在人肉看 `show ip bgp` 里更容易被检测。结合历史基线，异常检测可以在 BGP 收敛完成前触发告警。

**第五层：架构防御。** MANRS（Mutually Agreed Norms for Routing Security）是一套行业自律——定义前缀过滤策略、反欺骗、全球验证和协调四项行动。Peer-locking 是 ISP 之间的更高阶措施：两个对等方互相约束「你只宣告你有权宣告的路由，我只接受你有权宣告的路由」。这一层的防御力不是来自技术，是来自双边商业关系的杠杆。

五层不是替代关系。ROV 不部署就去搞 ASPA 没有意义——攻击者绕过路径验证只需要伪造起源。同样，有 ROV 没有 BMP 等于装了门锁但没装监控——你知道被入侵了（因为 ROV 阻断了），但不知道谁、什么时候、从哪里尝试过。

## §7.6 SR-MPLS/SRv6：标签分发从协议协商到控制器下发

传统 MPLS 的控制面是一锅协议杂烩：LDP 负责标签分发（逐跳），RSVP-TE 负责显式路径建立（逐流），BGP-LU 负责 inter-AS 标签交换。每一个协议维护自己的状态表，故障时同时查三张表排错——工程师要在 `show mpls ldp bindings`、`show rsvp-te session` 和 `show bgp vpnv4 labels` 之间人肉拼因果链。

Segment Routing 的核心简化不是技术性的，是架构性的：**路径状态从网络中移除。** 在 SR 模型中，路径由控制器计算后编码为 SID 列表（一段有序的指令序列），在入口节点一次性注入数据包头部。中间节点不再维护任何每路径状态——它只执行「看到 SID X → 执行动作 Y → 移到下一个 SID」的原子操作。排错时只需要检查入口节点的 SID 列表——不需要在每跳设备上追踪 LDP binding、RSVP-TE path state 和 BGP service label 的三重状态。

[E104] SRv6 运维简化的本质不是「用 IPv6」，是「路径状态从网络中消除」——排错锚点从 N 台中间设备缩减到 1 台入口节点。

SR-MPLS 和 SRv6 的迁移路径需要区分。SR-MPLS 重用现有 MPLS 数据面——标签仍然是标签，转发硬件不需要更换，通常通过软件升级即可激活。LDP 和 RSVP-TE 可以逐步退役，不必一次性关停。SRv6 则要求 IPv6 基础设施完整就绪，每个 SID 是 128 位 IPv6 地址，封装在 Segment Routing Header 里——硬件需要支持 SRH 处理和 uSID 压缩（将多个 SID 压缩进单个 128 位地址）。迁移的逻辑通常是一步一步的：先 SR-MPLS 替代 LDP/RSVP-TE（低风险，快速受益），再评估 SRv6 的额外价值（网络编程能力、端到端 IPv6 统一）。

但运维简化有一个被低估的代价：控制器变成了新的单点。在传统 MPLS 里，LDP 是分布式的——每个路由器自己决定标签绑定。在 SR 里，控制器负责计算和分发路径——控制器挂了意味着新路径无法建立。这不是说 SR 更脆弱，而是说脆弱性从「状态分散但有冗余」变成了「状态集中但可备份」。两种风险形态不同，需要不同的防护策略。

## §7.7 意图驱动网络（IBN）：从定义配置到定义意图

Perlman 在《Interconnections》第 17-18 章中批判了一种她称为「皇帝新衣」式的行业现象：用新术语包装旧概念，让买家以为自己买到了革命性的东西，实际上买到的只是旧方案的重新命名。IBN 面临的质疑就是：它是范式转移，还是 SDN 的重新包装？

区别在于自动化做的是「人写逻辑→机器执行」，IBN 理想中做的是「人声明意图→机器推导逻辑→机器执行→机器持续验证意图是否仍被满足」。多了两步：推导和持续验证。这两步才是 IBN 的硬核。

当前生产中的「IBN」部署大多停留在第一步和第三步之间——自动化配置生成——第二步（意图到策略的自动推导）仍然高度依赖预定义的模板和规则库。NetBrain 的路径是从「解码现有网络的设计意图」入手——这是一个务实的切入点：先搞清楚你现有的网络到底在干什么，再谈自动化。Cisco 的 IBN 实现强调持续验证（assurance）——配置改完以后持续比对实际网络行为与意图的差距。

RCr Wireless 2026 年 7 月的一篇产业分析给出了冷静的判断：「完全自主、自愈的网络仍然是 aspiration 而非 reality」，「vendor marketing 会让你相信 IBN 已经是现代网络的table stakes，但实际生产部署仍然有限。」

这不意味着 IBN 是骗局。意味着 IBN 的交付路径不是「革命在一夜之间完成」，而是三段闭环的自动化程度梯度提升：先自动化验证（Batfish 式配置建模比对意图），再自动化排错（Agent 推理根因+生成修复方案），最后自动化意图推导（Agent 从自然语言意图生成配置方案）。前两步已经在生产环境运行，第三步尚在探索。

## §7.8 SASE 与自智网络 L4

SASE（安全访问服务边缘）和自智网络 L4 是两个不同问题的答案。SASE 回答的是「分散的用户怎么安全接入分散的应用」——把网络和安全收敛到一个云交付平台上。自智网络 L4 回答的是「网络能不能自己管理自己」——基于实时数据做自调整预测和部署。

两者在运维层面有一个共同的暗线：缩小人类的操作面。

SASE 把分支机构的运维从「管盒子」变成「管策略」——SD-WAN 设备零接触部署，安全策略由云平台统一推送，本地不需要网络工程师。自智网络 L4 则更进一步：MasOrange 在 2025 年 5 月成为欧洲首家部署 L4 自智网络的运营商，AI 助手和 Agent 协同处理从故障检测到自愈的完整闭环。Orange 集团的目标明确——维护团队专注于复杂问题，AI copilot 和 agent 接管日常基础运维。

TM Forum 定义了自智网络五级成熟度，L4 的定义是「基于实时数据做自调整预测和网络实现」。截至 2025 年中，14 家运营商宣布了 L4 部分场景的目标，但实际全面部署仍然少数。Huawei 的博客中提到「运营商采用 L4 自智网络已经看到 real action」——但在产业分析中，「action」和「production at scale」之间仍然有明显距离。

Gartner 的 SASE 预测更为具体：「到 2028 年，50% 的新 SASE 部署将基于单供应商 SASE 平台，高于 2025 年的 30%」。这个趋势的运维含义是：运维对象从「多种设备的协作配置」向「单一平台的策略管理」收敛。收敛本身不一定降低复杂度——它只是把复杂度从设备 CLI 搬到了平台 GUI，而平台 GUI 的抽象层如果设计不当，反而会制造更隐蔽的配置错误。

## §7.9 AI Agent 替代人工运维：本章的核心重构

这一节是本章的论点核心。不是「AI 辅助运维」，是 AI Agent 正在系统性接管配置-验证-排错三段闭环中的后两段——而第一段中人的角色也在从「敲命令的操作者」变为「定义意图的决策者」。

证据链已经足够硬：

Azure Networking NOA Agent 投入生产两年，光纤故障检测时间缩短 60%，修复时间改善 25%。Cisco AgenticOps 30 天内执行 165,000+ 次自主运维工作流，覆盖园区、分支、工业、数据中心和服务提供商五个场景域。Gartner 预测：用于基础设施运维的 Agentic AI 将从 2025 年的不足 5% 增长到 2029 年的 70%。

但「替代」不是「消灭」。Agent 替代的是「在 CLI 里敲命令、在 show 输出里找线索、在脑海里跑控制面模型」这三个高度流程化的认知行为。工程师没有消失——他们转向了两个新角色：**意图定义者**（写 Agent 能理解的运维意图规范）和**最终裁判者**（在 Agent 不能自主决策的边界案例上做判断）。

关键约束不是 AI 的技术能力。三个硬约束决定了 Agent 能走多远：

1. **跨域遥测的完整性。** Agent 的诊断能力只和它能看到的遥测数据一样好。如果网络里还有设备只支持 SNMP polling 不支持 streaming telemetry，Agent 的感知就是滞后和残缺的。
2. **控制器级执行层的可审计性。** Agent 推配置进生产之前，必须有一条可审计的变更链——谁授权的、基于什么证据、影响范围多大、回滚方案是什么。这不仅是安全问题，是组织信任问题。
3. **政策护栏的硬度。** 什么情况下 Agent 可以自主执行变更？什么情况下必须等人批准？这个边界不是技术定义的，是由组织的风险承受能力和 SLA 合同定义的。

这三个约束没有一个纯粹是技术问题。它们说明了一个更深层的真相：Agent 替代人工运维的速度瓶颈不在模型能力，在运维组织自身的制度成熟度。

CableLabs 的 NetLLM、中国移动的 AI Agent 运维框架（draft-chuyi-nmrg-ai-agent-network）、Ericsson 的 AI Agent 网络架构白皮书——都在描述同一个走向。不同的实现路径，同一个收敛方向：运维决策的主体从人变成 AI Agent，而人的工作从「执行」变成了「治理」。

PARC 综合征在这里有一个精准的复现风险：**在不重新设计破碎流程的情况下，用 AI Agent 自动化这些流程——结果是更快地执行错误操作。** FirstPassLab 的分析指出，成功的团队「不是最先自动化一切的团队，而是在扩大自主权之前先结合跨域遥测、控制器级工具访问和硬政策护栏的团队」。这是运维 Agent 的最小可行信任基。

## §7.10 为 AI Agent 设计运维意图

如果 §7.9 的论点成立——Agent 正在接管验证和排错、改变配置端的角色——那么下一个问题就是：怎么给 Agent 写意图，让它可以理解和执行？

人的运维意图通常是模糊的、隐含的、依赖上下文补全的。跟同事说「把视频流量优先走 path-A」，隐含了同事知道 path-A 是哪个接口、视频流量的 ACL 怎么配、优先级是指 local-preference 还是 DSCP marking。Agent 没有这个隐含上下文——或者说，Agent 的上下文来自它被配置的遥测数据源和知识库，跟人脑里的经验不是同一套。

为 Agent 写意图的三个约束：

**可验证性。** 意图必须是可被自动观测验证的——「优先级高的流量延迟应该更低」是意图，「当路径 A 可用时视频流量走 A」也是意图，但前者需要翻译为可观测指标（延迟阈值、测量方法、采样频率），否则 Agent 无法判断意图是否被满足。

**边界闭合。** 意图必须明确什么是「不在范围内」的——Agent 不会自动知道「这个意图只管视频流量，不管语音流量」。人的意图天然有边界（说「视频走 A」意味着语音走默认路由），Agent 的意图必须有显式边界定义。

**可逆性。** 每个 Agent 执行的变更必须有可自动触发的回滚条件。不是「出了问题人工回滚」，而是 Agent 自己能检测「变更后延迟反而上升超过阈值」并自动回滚。这不是技术难题——是意图设计时就必须写入的约束：告诉 Agent 什么叫失败。

Batfish 的验证方法是这个方向的一个先锋：在不接触生产网络设备的情况下，用配置文件构建全网络控制面模型，模拟所有可能的转发路径后逐条比对意图断言（「VLAN 100 不能路由到 VLAN 200」「所有 BGP 会话必须 established」「prefix X 只能从 peer Y 接收」）。Agent 可以在变更推入生产之前，在这个镜像里完整验证所有意图——如果验证不过，变更门禁直接拒绝。

这听起来像 CI/CD for networks。本质上它就是。代码通过 CI 管道验证推入生产，配置通过 Agent 验证管道推入网络。区别在于——网络的「测试环境」就是生产网络的控制面模型，因为网络的正确行为不仅取决于单设备配置，取决于所有设备配置之间的交互。

Gartner 预测 Agentic AI 运维的采纳曲线时没有说的是：到达 70% 采纳率的前提不是模型变强，而是意图规范体系的建立。你能把「这个网络应该怎样运作」写成可被 Agent 自动验证的声明式意图——你就能用 Agent 做运维。你不能——Agent 再强也没用。运维 Agent 的瓶颈不在 AI，在表述运维知识的语言。

## 经典深层注

**Zhang-Bartell 最被低估的贡献**不是 BGP 的某个具体配置技巧，而是「网络身份前置」这个方法论决策——把「这个网络是干什么的」放在「这个网络怎么配」之前。二十年后回头看，这条原则的价值不是被证实了，是被遗忘后反复被重新发现了。每一个「我们照着 SP 模板配了企业网然后出了事」的故障报告，都是这条原则的一次免费复证。

**De Ghein 的运维管线**提出了一个被行业选择性忽略的框架：配置、验证、排错是三个等权重的环节，不是一个主环节（配置）和两个辅助环节。大多数组织在配置上投入 80% 的精力（写方案、画拓扑、出配置模板），在验证和排错上合起来投入 20%。AI Agent 注入后，这个分配比例被颠覆了——不是因为 Agent 需要更多精力，而是因为「验证」第一次有了独立于「排错」的生产价值。

**Perlman 的术语批判透镜**放到 IBN 上仍然锋利。不是问「IBN 是什么」，是问「IBN 声称解决的问题，在 SDN 和网络自动化里有没有被解决过」。如果意图推导依赖预定义模板，如果持续验证依赖固定阈值，那么 IBN 中的「I」是被过度承诺的——它没有真正从意图推导，它只是把意图映射到了模板。这不妨碍当前 IBN 实现的生产价值，但它限定了「IBN 和 SDN 的区别到底在哪」这个问题的诚实答案。

**「构建者不能验证自己输出」原则**在运维领域的复现是最新但最深刻的发现。Meta 的 AI Agent 验证实验之所以有 37pp 提升，不是因为 Agent 的模型能力碾压人类，而是因为「独立的外部视角」结构性地消除了确认偏误。这和 deep-research skill 的 Challenger Gate 同构，和 shopping-claim-verify 的 Phase Gate 同源。不同领域，同一结构：审计和构建必须由两个独立实体执行。

---

*运维设计揭示的是网络在组织中的真实位置——不是买了什么设备，是出了事怎么响应。但组织不是凭空产生的。网络运维的组织方式、决策权限、工具选择，全都是特定制度环境的产物——不是「最优方案」赢了，是被特定制度允许的方案活了下来。* 下一根骨头回答的不再是「怎么运维」，而是「谁决定了怎么运维」——以及那个「谁」在历史上长什么样。
