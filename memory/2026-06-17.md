## 📰 晨会金融速递 cron 静默失败（误报成功）

**现象**：20:00 速递 cron 触发，heartbeat 报「completed successfully」，但实际 0 输出，微信没收到速递。

**取证**（`logs/unified-2026-06-17.log` + `myagents cron runs --json`）：
- cron 会话 `33015c41` 调 `deepseek-v4-pro`，API 返 `400 模型不存在，请检查模型代码`，`is_error:true`、`output_tokens:0`、content=null
- 6/15、6/16 两天的 run 都有完整 4 条速递，今天突然断 → 非首次，是新鲜故障
- `myagents cron list` 显示 `lastRunOk:true`、heartbeat 报 success —— **框架误报**

**根因（已确认部分）**：mino agent 的 channel override model 是 `deepseek-v4-pro`（config.json `agents[].channels[].overrides.model`）。cron 会话 boot 显示 `model=glm-5.2`（session 默认），但实际 query 用 override 的 `deepseek-v4-pro`。该模型今天在 provider 侧返 400「模型不存在」——疑似 provider 改名/下架，未深查。

**误报坑（新教训）**：cron 框架的「成功」只代表调度+sidecar 执行成功（HTTP 200），**不代表 AI 产出了内容**。AI 报 400/0 输出时，`lastRunOk` 仍为 true、heartbeat 仍报 success。排查 cron「没执行」必须看 `cron runs --json` 的 `content` 字段是否为 null，不能信「completed successfully」。

**处置**：
- 当晚手搓 4 条速递（陆家嘴论坛央行 6 项政策为主：利率走廊收窄 / 数币 CBETS 上线 / 境外央行回购工具+离岸人民币交易试点 + 银行业数据安全罚单激增），直接微信发给汤姆
- 修复方案（已向汤姆提议，待确认）：把速递 cron 模型从 `deepseek-v4-pro` 切到 `glm-5.2`（当前会话在用，正常）

**待办**：汤姆确认后改 cron 模型并验证；有空查 deepseek-v4-pro 为何 400（provider 改名？）。

## 🧭 探路：cron 静默失败的发现路径
**出发点**：heartbeat 报速递「completed successfully」，但凭直觉觉得没看到内容。
**过程**：先 `cron runs --json` 看 content → 最新 run 是 null（前两天有内容）→ 拉该 session 的 unified 日志 → 一眼看到 `API Error: 400 模型不存在` + `is_error:true`。
**教训**：cron 的「success」不可信，`cron runs --json` 的 content 字段才是真相。heartbeat 是调度层信号，不是内容层信号。
