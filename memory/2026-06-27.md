### 核心验证

Challenger Gate 是 v2 最重要的单项收益。量子计算测试中父 Agent 在 6 个发现上犯了系统错误——把模拟当硬件(IQM 1000x)、把单人观点当共识(Hangleiter)、把实验室实验当商业产品(QuEra 96 LQ)、把"非首次"当"首次"(Google表面码)、把并列关系当主次(人才vs硬件)、把预印本当同行评审(IonQ qLDPC)。全部被 Challenger 否定性搜索纠正。

**"构建者不能验证自己的输出"被硬数据证实。**

## 🧭 探索：从概念到 v2 验证的完整 build 流程

**出发点**：汤姆一句话"如何打造搜商无限的 agent"。没有 spec，没有需求文档，只有概念方向。

**过程**：
1. 先出架构设计（六层模型 + 四种能力乘积）→ 框架验证前不动手
2. 读现有 skills 学习模式（claim-verification/shopping-claim-verify/cognitive-gap）→ 先读再写
3. v1 写完后立刻实测（AI芯片）→ 不猜测是否好用，看实际表现
4. 实测后保留成功部分 + 找可优化点（分母陷阱、验证矩阵格式、一轮收敛说明）
5. 外部技术调研（arXiv 论文 + Zylos 调查）→ 确认优化方向不是闭门造车
6. v2 全增强落地（P0+P1+P2+工具层）→ 一鼓作气，不拆分迭代
7. 第二次实测（量子计算）→ 核心发现：Challenger Gate 发现 6 项错误
8. session-archive 存档 → 设计密度持久化

**教训**：
- "先出架构再动手"在 AI agent skill 构建中同样成立。架构错了后面全废
- 实测 > 猜测。AI芯片测试暴露了分母陷阱，量子计算测试暴露了 Challenger 的价值——两次测试发现的东西完全不同
- Challenger Gate 是"免费午餐"——从 shopping-claim-verify 移植几乎没有工程成本，但收益巨大
- "一鼓作气全落地"比"逐步迭代"更适合 skill 构建——skill 是指令文件不是代码库，分批次改动容易导致版本不一致
- 47KB 的 skill 文件(5个文件)不算大——关键不是体积是结构。每个 reference 文件解决一个独立问题，不堆到一个文件里
