## 写书工具 MVP 建成

从一次"经典要不要先读完"的追问开始，一整条主张驱动写书管线从零建到跑通。

### Canon Mapper Skill
- 五层管线: L0 领域识别 → L1 经典发现 → L2 经典获取 → L3 骨架提取 → L4 框架映射 → L5 搜索方向
- SQLite claims.db (7 表): classics, classic_skeletons, claims, claim_chapters, claim_dependencies, framework_mappings, search_directions
- 3 个 Python 脚本: db.py (CRUD + migrate + affected), extract_toc.py, map_framework.py
- 参考文件: domain-taxonomy.md

### 集成打通 (P0)
- deep-research: Canon Mapper 集成模式——消费搜索方向→自动写入 claims
- claim-verification: Database Write Mode——验证并入库
- 三个 skill 互相引用一致

### 引用追踪 (P1)
- db.py migrate: 扫描 markdown 章节→提取 [DR00X] 引用→写入 claim_chapters
- db.py affected: 查主张被哪些章节引用→改主张时标受影响范围

## 🧭 探索: Canon Mapper + 主张驱动写书管线

**出发点**: 汤姆问"经典要不要先读完"→讨论"经典的价值不在内容在框架"→提出经典映射器概念

**过程**:
1. 设计 Canon Mapper 五层管线——模仿 deep-research 的结构但领域不同
2. 建 SQLite 数据库做主张持久化——选择 stdlib sqlite3 零外部依赖
3. 手工注册 10 本已有经典 (金融书 23 份参考文献中的核心 10 本)
4. 用米什金做冒烟测试——搜索目录→提取骨架→映射→生成搜索方向
5. 发现 gap (8 个)→ 生成 11 个搜索方向 → 用 deep-research 消费
6. 用 Challenger Gate 验证——证实了偏差确实存在
7. 打通三个 skill 的数据流——从手动到自动
8. 建 migrate + affected 实现引用追踪

**教训**:
- Skill 之间通过 claims.db 互操作比通过文件传递可靠得多——数据格式一致、校验在写入时执行
- Challenger 的价值不是"偶尔发现错误"——是每次都能发现
- 引用追踪 (affected) 是写书工具的最被低估的功能——不知道改了主张影响哪些章节，就不敢改
- P0/P1 的改动非常小 (几个 SKILL.md 段落 + 两个 db.py 命令)，但效果巨大——全线从"手动"变成了"自动"
- P2 的渲染层 (主张图→markdown) 是真正的硬骨头——涉及自然语言生成和结构一致性
