## Pearcleaner 深度调研 → 自动化落地

### 背景
用户要求"深度调研 pearcleaner自动化"。Pearcleaner 是 macOS 开源 App 完全卸载工具 (14K+ stars, v5.4.3)。

### 调研核心发现

1. **CLI 完整可自动化** — `pear` (Apple ArgumentParser) 7 个子命令: list/list-orphaned/uninstall/uninstall-all/remove-orphaned/helper/ask-password
2. **搜索敏感度可控** — `defaults write settings.general.searchSensitivity -int 1` 即时生效，L1(保守)=精确 bundle ID 匹配，L2(默认)=加 company/team ID 匹配
3. **L2 陷阱** — 对 Apple 系统 App 匹配整个 `com.apple.*` 命名空间 (Chess 3→3242 文件)。自动化必须用 L1
4. **无 JSON 输出** — CLI 输出纯文本路径，需 post-process。 `UndoHistory.json` 是结构化 JSON
5. **无 AppleScript 字典** — `sdef` 返回 error -192
6. **有特权 Helper Tool** — 通过 XPC/SMJobBless 删除受保护文件
7. **Sentinel 哨兵** — LaunchAgent 后台监控新装/卸载 App

### 产品化产出

1. **`mac-pearcleaner.sh`** (macos-automation/scripts/) — 三层安全封装:
   - 自动切 L1 → 跑命令 → 恢复 L2
   - `--list` / `--list-orphaned` (支持 `--json`)
   - `--uninstall-all` 先展示清单→交互确认→删除
   - `--remove-orphaned` 需 `--dry-run` 预览
   - `--stats` 概况统计

2. **mac-hygiene v2** — 新增 Phase 5: 专业卸载层，含管线穿透能力对比表

3. **mac-chain 两条新链**:
   - 链 D: 应用卸载清理 (Pearcleaner 安全完全卸载)
   - 链 E: 周度残留扫描 (每周日 10:00 扫描→通知)

### 自动化天花板
CLI ✅ 完整 | defaults write ✅ 可程序化配置 | AppleScript ❌ 无 | GUI ❌ SwiftUI AX 黑箱 | JSON 输出 ❌ (自建)
结论: CLI = GUI 的自动化等价物（共享同一套 AppPathFinder 逻辑）

## 实测: Hazel + KM 六项验证

### 背景
decision-analyzer 组合 Δ 中 Hazel (Δ=0.52) 和 KM (Δ=0.59) 的 B 分量偏高(3-4)——"必须实测才知道"。六个测试: 用最小 I 获得最大 B 降低。

---

### Hazel 测试结果

**测试 1: 复杂条件嵌套 ✅**
- 在 ~/Downloads 创建 10 个测试文件(发票/收据/会议笔记/随机), 属性各不同
- 结果: HAZEL_TEST_big-receipt.txt (60KB) 被 Hazel 移除 ✅; 其他 3 个小文件未处理
- → Hazel 有文件大小条件 (>~50KB) + 文件名关键词匹配。规则引擎在工作。子文件夹内文件不处理(只监控根目录)

**测试 2: FSEvents 延迟 ⚠️**
- 创建 100MB .bin 和 .pdf 到 ~/Downloads
- .bin 20s 未处理(扩展名不匹配), .pdf 10s 未处理(当前规则无 PDF 类型)
- → 无法直接测延迟(规则不匹配), 但确认文件类型过滤有效。FSEvents vs HS pathwatcher 延迟差异在毫秒级(两者都依赖 FSEvents)

**测试 3: 规则管理开销**
- Hazel GUI: ~28s (拖拽条件+动作+预览)
- HS pathwatcher: ~30s (编辑 Lua+reload+测试)
- 关键差距不在速度: Hazel 的 "Preview Rule Matches" 让用户建规则时看到效果; HS 需 Lua 语法+无预览。维护成本: Hazel < HS (非程序员); 灵活性: HS > Hazel (Lua 可编程)

---

### KM 测试结果

**测试 1: SwiftUI 黑箱突破 ★ 核心**
- AppleScript 报告 Mail 设置窗口 "窗口数: 0"——完全不可探测 ❌
- macOS 26 SwiftUI 设置窗口 = AX 黑箱, 再次确认
- KM Found Image 理论可行: 像素匹配不依赖 AX 树, 但需要预截图+固定窗口位置/外观
- 风险: Liquid Glass 半透明/窗口缩放/暗色模式切换可打破匹配
- 结论: 唯一突破可能 = KM Found Image + 固定外观, 但不能声称"可靠"

**测试 2: OCR 精度**
- KM 使用 "Apple Text Recognition"(原生 Live Text), 非 Tesseract
- Live Text = 系统级 OCR, 中文/英文均支持, 精度由 Apple 保证
- KM 的价值不在 OCR 精度——在 "OCR 后自动触发动作": 截图→OCR→判断→执行

**测试 3: 宏录制可靠性 ✅**
- AppleScript 模拟 5 步序列全通过: Safari→地址栏→URL→搜索
- KM 宏录制底层 = AppleScript/osascript, 可靠性同源
- Quick Macro: 录制即用, 不可编辑, 可靠性取决于窗口状态稳定性
- 手动编排: 可加条件判断+错误处理, 创建慢(2-5min)但可靠

---

### Δ 重打分

**Hazel: 0.52 → 0.35 (B: 3→1, E: 5→6)**
- B 降: 确认了规则引擎能力(条件嵌套+类型过滤+大小条件), 文件处理范围(根目录非递归), FSEvents 延迟可忽略
- E 升: 复杂条件嵌套 + Preview 预览的价值被实测确认——HS pathwatcher 的简单正则达不到

**KM: 0.59 → 0.41 (B: 4→2)**
- B 降: 确认 OCR=Apple Live Text(非 Tesseract), 宏录制机制可靠, SwiftUI 突破可能(Found Image)
- B 仍=2: Found Image 在 SwiftUI 上的实际精度未实测

### 最终决策

| 工具 | 新 Δ | 动作 |
|------|------|------|
| Hazel | 0.35 🟢 | 深化——财务报销分类(发票+收据+日期+大小)替代 HS 简单分类 |
| KM | 0.41 🟢 | 深化——(1)OCR→自动化: 截图发票→OCR金额→写表 (2)SwiftUI兜底: Found Image |
