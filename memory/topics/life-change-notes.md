# Life Change Notes — 生活变更记录系统

> 2026-06-30 创建。起源：汤姆将工作中监管 change notes 的中间产物概念迁移到生活整理——被 deep-research 调研验证并深化。

## Quick Reference

- **What**: 生活领域的"变更记录"中间产物——在工作基线习惯的基础上，对生活中新加入/改变/移除的物品、习惯、规则产生追踪记录。核心洞见：**工作 change notes 靠问责链运行（监管→领导→同事→合规），生活天然缺乏这个结构——所以照搬 event-based 必然失败。**
- **Status**: active (last worked on 2026-06-30)
- **Key Insight**: 问题不是"触发器怎么设计"，是"生活没有问责结构"。三种可行替代：物理问责（容量=触发器）、对话问责（bot/AI/人=触发器）、决策问责（预测vs实际=触发器）。Phase 2.5（即时关键词草稿 + 周 review 展开）可能是最优解本身，不是过渡阶段。
- **Architecture**: 三层——Baseline（基线习惯+物品管理规则）→ Change Notes（变更流，追加式记录）→ Review（周 review 提炼+更新基线）
- **Blockers**: 零外部问责的生活环境——这是根本性约束，不是技术问题
- **Pending**: (1) 汤姆定 2-3 个基线习惯 (2) 选一种问责模式试点 (3) 写 life-baseline.md + life-changenotes.md 初始文件

## Concept Definition

将工作中"监管变更→change note→流程更新"的三段式信息处理模式迁移到生活领域。工作中 change notes 依赖硬性嵌入（监管推送+岗位职责+同事互审+领导检查+合规证据），生活场景天然缺乏这些问责结构。因此不能照搬 event-based 模式，需要重新设计触发和问责机制。

核心区分：
- **Review-based**（定期审查型）：所有现有生活 OS 系统的做法——季度/每周回头看，适合低频高判断的变化
- **Event-based**（事件驱动型）：工作中的模式——变化发生时立即记录，依赖外部强制触发器
- **关键洞察**：真正区分不是 review vs event，是**有问责 vs 零问责**

## Architecture

```
┌─────────────────────────────────────────┐
│              问责层（外挂）               │
│  物理问责 │ 对话问责 │ 决策问责           │
│  (容量=触发) (bot/人问) (预测vs实际)      │
└────────────────┬────────────────────────┘
                 │ 触发记录行为
                 ▼
┌─────────────────────────────────────────┐
│  Baseline（基线）                        │
│  · 2-3 keystone 习惯（MVR 级别）          │
│  · 物品管理规则（OIOO 适用类别）           │
│  · 生活规则清单                           │
└────────────────┬────────────────────────┘
                 │ 变化发生
                 ▼
┌─────────────────────────────────────────┐
│  Change Notes（变更流）                   │
│  · 物品类：[日期] 入/出 [物品] 原因 [替代] │
│  · 习惯类：[日期] 始/止 [习惯] 触发 [状态] │
│  · 规则类：[日期] 新规 [描述] 取代 [旧规]  │
│  · 追加式记录，不修改历史                   │
└────────────────┬────────────────────────┘
                 │ 周 review 提炼
                 ▼
┌─────────────────────────────────────────┐
│  Review（提炼层）                         │
│  · 每周日 30min                          │
│  · 检查基线执行率（Win Rate >70%）        │
│  · 展开关键词草稿为完整 note              │
│  · 评估：哪些变化试验成功/失败/待观察      │
│  · 决定：是否更新基线                     │
└─────────────────────────────────────────┘
```

## Three Accountability Models

### A. 物理问责：容量 = 触发器
- 适用：物品管理
- 机制：给每个物品类别设固定空间容量 → 容量满必须决策 → 决策时刻自然产生记录需求
- 问责源：物理空间不可忽略（衣柜塞不下了你必须处理）
- 优势：不需要动力、不需要记忆、不需要外部提醒

### B. 对话问责：bot/AI/人 = 触发器
- 适用：习惯变更、规则变更、生活决策
- 机制：固定对话节奏（每天/每两天），Claude 或 bot 主动发起"最近有什么变化？"
- 问责源：对话约定——"有人在问"和社会问责同构
- 优势：借用了"回复消息"的已有行为模式，不建新习惯

### C. 决策问责：预测 vs 实际 = 触发器
- 适用：重要决策
- 机制：做决定时写下预计结果+验证时间 → 到期强制对比 → 差异产生 change note
- 问责源：好胜心——"不想让自己的预测落空"
- 优势：不依赖外部，利用内在动机

## Key Design Decisions

1. **不照搬工作 event-based** — 生活中没有监管推送/领导检查/合规处罚这条问责链。纯 event-based 在零问责环境中不可持续。被 reject 的方案：完全复刻工作中的"事件发生→立即记录"模式。
2. **Phase 2.5 可能是终点** — 即时关键词草稿 + 周 review 展开，可能不是"过渡"，而是生活场景下的最优解。它承认 event-based 不可靠，把捕获成本降到最低（一个词），把整理留给 review。
3. **问责源外挂，不依赖意志力** — 不为"我怎么才能记得做"设计方案，而是为"什么外部力量会让这件事自然发生"设计方案。
4. **按变化类型定最小记录格式** — 物品/习惯/规则各有 2-3 行的预置模板，降低每次记录的脑力成本。被 reject 的方案：通用格式（要么太抽象要么太复杂）。

## Minimum Executable System

### Phase 1: 定基线（一次性）
- 选 2-3 个 keystone 习惯（MVR 级别：最差的一天也能做）
- 明确物品管理的 OIOO 适用类别
- 产出：`memory/topics/life-baseline.md`

### Phase 2: 周 review（持续）
- 每周日 30min：检查基线执行率 + 展开关键词草稿 + 写完整 change notes
- Change note 格式：类型 + 一句话描述 + 触发 + 决策 + 状态
- 产出：`memory/topics/life-changenotes.md`（追加式）

### Phase 2.5: 即时草稿（习惯稳定后加入）
- 手机备忘录/微信发给自己：只记一个关键词或一句话
- 不分类、不评估、不格式——只保存记忆
- 周 review 时展开

### Phase 3: 问责源接入（按需）
- 选一种问责模式试点——物理问责最容易开始（不需要新技术/新习惯）
- 对话问责可随时启用（你已经在跟 Claude 对话）

## Known Limitations

| # | Limitation | Severity | Mitigation |
|---|-----------|----------|------------|
| 1 | 生活天然零问责——任何系统都依赖自我执行 | 根本性 | 三种问责模型外挂。接受不完美：80% 周完成率即成功 |
| 2 | 基线习惯的选型和坚持是前置条件 | blocker | 先选 2-3 个 MVR 级别（最差一天也能做），不追求理想状态 |
| 3 | 没有现成 event-based 模板可复制 | major | 从 review-based 开始，Phase 2.5 降低捕获成本，逐步逼近 |
| 4 | 中文社区无个人整理方法论讨论 | minor | 英文源（Git-based Life OS/OIOO/MVR）+ 工作管理经验迁移 |
