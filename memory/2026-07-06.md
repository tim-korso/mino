## 🧭 探索：从隐喻到农场管理系统的设计

**出发点**：汤姆用"希望麦田"的比喻描述他想做的事——多 Agent 像麦田一样分布在不同领域，各自深耕，他做守望者。概念很美，但需要把隐喻落地成可操作的系统架构。

**过程**：先读已有的 goal-loop 架构（五模块+四检测器+声明式验证）和 deep-research v2（七层管线+Challenger Gate），确认底层灌溉系统已经就绪。然后识别缺口：不是缺田，是缺农场管理器——portfolio view、资源分配、收割标准、跨田授粉、操作节奏。从 life-change-notes 的问责模型获得启发（周 review 节奏可以跨系统对齐）。最后用声明式 YAML 做田定义格式（K8s 风格），用 Markdown 仪表盘做巡田看板，用 slash command 做操作入口。

**教训**：
- 隐喻是用户的 spec。汤姆没说要"建一个 portfolio management system"，他说了"希望麦田的守望者"。但这两个是同一个东西——隐喻编码了架构需求。
- 已有 80% 的基础设施。Goal Loop + KB + MCP tools 就是灌溉系统。新东西不是重写底层，是在上面加一层管理规则。
- 操作节奏 > 自动化。仪表盘是手动刷新的 markdown，巡田是手动读的。这看起来"不够自动化"，但对汤姆来说——他需要的不是自动报表，是每周 15 分钟亲手触摸每块田的状态。自动化适合耕种环节，不适合管理判断。
- 两套周 review 对齐。生活 change notes 周 review + 农场巡田同一天做，两个习惯互相锚定。这是从 life-change-notes 设计中学到的——孤立的新习惯难以持续，锚定到已有节奏才能活。
