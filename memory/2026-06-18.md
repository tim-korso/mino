## 🧭 探索：克隆音色实时对话 — 从端到端到级联式 (22:00-03:00)

**出发点**：豆包端到端实时语音体验好但音色固定，想用自己上传的音频声音做实时对话

**过程**：
1. 豆包端到端跑通(二进制V1协议，三件套鉴权，24kHz float32 PCM)
2. MiniMax Realtime 纯JSON协议一次通过，但克隆音色 `nana_v2_20260617` 报 2042"you don't have access"
3. 确认：**端到端商业API(豆包/MiniMax/GPT-4o realtime)都不支持克隆音色进实时对话**——行业物理限制(低延迟 vs 克隆音色二选一)
4. MiniMax 声音克隆本身能做(转 ALAC→mp3 后上传成功，`volcengine-audio` SDK 的 protocol 帧一次通过 ASR)
5. 级联式(ASR→LLM→TTS)三段冒烟全通：豆包ASR(SDK帧) + DeepSeek + MiniMax TTS HTTP(克隆音色)
6. 麦克风实时录音 ASR 识别待排查(文件模式通过，4s实时录音不出字)

**教训**：
- 端到端低延迟和克隆音色是行业物理边界，商业API全部二选一
- 手搓豆包二进制协议 flags 两次失败(0b0000 vs 0b0001)——用 `volcengine-audio` SDK 一次通过。**手搓二进制协议靠猜不可靠，SDK/参考实现是正路**
- ALAC 编码的 m4a 会被平台拒绝("参数错误")，ffmpeg 转 mp3 后立刻通过
- WS 事件字段 string vs int 不能凭文档猜，要看实际返回
- MiniMax 克隆音色 7天不用会删；voice_id 命名规则[8,256]字母开头
- 级联首音延迟 ~1.5-2s(比端到端 ~300ms 高一截)，这是"要声音"的代价
