### 云盘链接自动验证系统 (dl-validate)

从"链接验证必须亲自试"到"一个 POST 请求 500ms 判死活"。发现了夸克/阿里/115 的匿名 API——均无需登录即可判断链接状态。

- 夸克: `POST drive-pc.quark.cn/.../token` → status:200=活, status:404=死 (41012=取消, 41019=过期)
- 阿里云盘: `POST api.aliyundrive.com/v2/.../get_by_anonymous` → 无code=活, ShareLink.Expired=过期
- 115: `GET webapi.115.com/share/snap` → errno=4100012=需密码, errno=4100033=违规
- UC: 同夸克 API (阿里系共享后端)
- 123/天翼: HTTP 级别信号 (404/302)
- 百度: HTTP 404=确定死, 200=可能活

迅雷 Nuxt SSR 盲区——有效和失效链接的 Nuxt 状态完全一样（参数一致，都是 SSR 空壳），share status 纯客户端异步拉取。移动云盘/蓝奏云/城通同理。

## 关键教训

1. 匿名 API 发现是可迁移技能——夸克 API pattern → UC (同阿里后端)
2. "构建者不能验证自己输出" 再次验证——cognitive-license 发现了 C018/C022 的问题
3. 扩散模型≠定位的元规则延伸到网页解析——SSR 盲区 = 静态分析不可达 = 需要 JS 执行
4. share-sniffer 的价值命题是给人用的 GUI，不是给 Agent 调用的 API

## 🧭 探索：云盘匿名 API 发现

**出发点**: C018 "链接验证必须亲自试" —— cognitive-license 判了 DIRECTION，但这是基于当时知识边界的判断。

**过程**:
1. 夸克: 发现 HTTP 层盲区 (11,589 bytes 完全相同) → 搜开源项目 → 发现 token API → 实测: 41012/41019/200 三种状态清晰可辨
2. 阿里云盘: 阿里云官方文档有 `getShareLinkByAnonymous` API → 实测: 三种状态码完全区分
3. 115: GreasyFork 脚本提示 → 猜 URL 参数 → 命中 `webapi.115.com/share/snap` → 实测: errno 精确区分
4. UC: 猜同夸克后端 → 实测: 完全正确
5. 迅雷: Nuxt 状态执行 → SSR 盲区 → 参数对比 → 完全一样 → 堵死

**教训**: 匿名 API 发现不是运气——中文网盘普遍有"分享页预览"功能，背后就是不需登录的 token/info API。搜开源项目(share-sniffer/PanCheck/GreasyFork 脚本)比逆向工程 API 快 10 倍。**发现 API 的方法是找已有实现，不是猜端点。**
