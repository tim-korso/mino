## Goal Loop 设计还原 + Ethan 文章对齐

### 背景
汤姆分享了 Ethan 的公众号文章《用「造人」的视角设计 Agent 系统？这可能就是错误的原点》，问和 goal loop 的关系。核心论点：Agent 是意义先于存在——先有 Goal，先有委托关系，然后 Agent 才成立。不应该先「造人」再找事。

然后汤姆给了五张设计图—— Agent 收集信息与预测未来能力的设计探索。他发现记忆里找不到这部分内容——只有 alignment record 里一句「GoalLoop 收敛验证——娜娜 dual-model-mcp 已有原型」。

### 五图内容
通过 Qwen-VL-Plus 提取了全部五张图：

1. **七维能力矩阵** — dual-model MCP 的 7 项能力实测评估：单文档/交叉验证/自主搜索/收敛检测/置信度校准/多模型共识/持续监控
2. **增量知识库检测器** — WoW Diff(5★) → 主张漂移(3★) → 叙事突变(4★) → 矛盾轨迹(2★)，四层递进
3. **五个 Python 模块** — MonitoringPipeline / GoalLoop / ConsensusVerifier / QuestionBank / ClaimKnowledgeBase，全部持久化到 ~/.myagents/kb/
4. **增量知识库流程（场景4）** — Day 1..N: Flash×N→JSON→存库；Day 30: Pro 综合 → "本月政策趋势"
5. **K8s 式声明式验证目标** — 从「帮我验证5份文档」变为声明 desired state，系统持续 reconcile

### 产出
- **新 topic file**: `memory/topics/goal-loop.md` — 完整还原设计密度：架构/模块/检测器/流程/未来愿景/已知局限/未做优化/技术实现

### 关键发现
- **设计密度 vs 记忆碎片**：这五张图里的设计远超过 memory 系统里能找到的——alignment record 一句话不足以还原。问题在于「探索过程中产生的设计图没有回写到 topic file」。
- **qwen-vl-plus 能读图**：当前 session 的 deepseek-v4-pro 不支持图片，但用 DashScope API + qwen-vl-plus 成功提取了五张图的完整内容。以后遇到图片可以用这条路。

## Session 退出动作固定化方案

### 问题
几周后打开新 session，如何清晰定位已有的探索设计？需要：
- 知道发生了什么
- 产生了什么（topic files, code, DB）
- 原理和技术决策
- 未来也许能优化什么
- 没做的优化还有哪些

### 当前机制
- `/UPDATE_MEMORY` 命令 — 存在但侧重记忆维护，不是设计捕获
- 探路叙事（🧭 探索） — 每日日志里的三段式，但依赖 Agent 在任务完成后主动写
- topic file — 有模板但各文件密度参差不齐

### 缺口
- **无强制性**：Agent 可能跳过，topic file 可能不更新
- **无结构化**：探路叙事的密度不如设计图
- **无自动触发**：完全依赖 Agent 自觉

→ 见下方机制设计讨论
