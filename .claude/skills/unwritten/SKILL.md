# 潜规则判断引擎 v3 — Unwritten Rules Analyzer

> 不是告诉你"这个领域的潜规则是什么"——是教你怎么结构性地发现任何领域的潜规则。
>
> 核心定理：**潜规则是从正式制度的裂缝里长出来的。**
>
> **v3: 三 Agent 管线。** 主分析 → 信息不对称 Challenger（只看到数据层，不读到推理过程）→ 负空间检查（暴露框架本身漏掉的视角）。输出带许可等级。

## 快速使用

```bash
# CLI 直接问
python3 unwritten-rules.py "明天去见XX局的领导谈项目审批，有什么要注意的？"

# 指定域，加速搜索
python3 unwritten-rules.py --domain 招投标 "投标一个政府项目，同行说已经定了"

# JSON 输出（管线消费）
python3 unwritten-rules.py --json "新买的房子物业突然要收一笔未在合同里的费用"

# 对比模式：看框架 vs 裸 LLM 的区别
python3 unwritten-rules.py --compare "抖音账号流量突然下降"
```

## 竞争壁垒

| 维度 | 传统咨询/知识库 | ChatGPT/裸 LLM | **这个引擎** |
|------|---------------|----------------|-------------|
| 知识来源 | 预先积累的域知识 | 训练数据中的模式 | **实时搜索 + 框架推理** |
| 覆盖域 | 有限的、预设的 | 广泛但浅层 | **任意域——框架通用** |
| 分析深度 | 域专家级别 | 常识/故事级 | **结构性——追到制度裂缝** |
| 权力分析 | 有但不成体系 | 回避或泛泛而谈 | **合法伤害权·位置分析·规则切换** |
| 更新 | 依赖人工维护 | 训练数据截止日 | **每次查询实时搜索** |
| 核心能力 | "我懂这个领域" | "我见过类似模式" | **"我知道怎么结构性地拆解任何领域"** |
| 可迁移性 | ❌ 换域要换专家 | ⚠️ 不可控 | ✅ **框架放任何域都管用** |

## 五步分析框架

### Step 1: 定位正式规则
- 明文规定/法律/合同/约定是什么？
- 执行机制：谁执行？成本谁承担？
- **裂缝在哪？**（自由裁量空间、信息不对称、监督成本过高的区域、规则自相矛盾处）

### Step 2: 识别潜规则形态
- 裂缝里实际支配行为的替代规范是什么？
- 参与者按什么**利害计算**在行动？（不是"他们坏"——是给定约束下成本最低的选择）
- 系统性还是个例性？
- **合法伤害权**来源：谁有自由裁量权？谁能用正式规则赋予的空间伤害/恩惠他人？

### Step 3: 判断规则切换信号
- 潜规则处于扩张/稳定/收缩？
- 切换信号（吴思四级模型）：正式→潜规则的成本阈值？潜规则→血酬的威慑力下降？
- **运动式执法**信号（边界突然硬化）？
- **政治罪**信号（上级把普通事件升级定义来重新划定权力边界）？

### Step 4: 位置分析
- 用户在制度结构中的位置？这个位置决定什么约束？
- 博弈对手是谁？他们的约束是什么？
- 结构性压力来自哪个方向？（黄仁宇的七面镜子）

### Step 5: 行动建议
- 可行选项集合（给定约束下实际可用的，不是"理想选项"）
- 每个选项的**代价**（戚继光困境：不是好和坏之间选——是"这个代价"和"那个代价"之间选）
- **最低有效剂量**策略
- **止损条件**

## 铁律

1. 不假设制度有效——不提供"举报""走法律途径"除非确认该制度在此情境下确实有效
2. 不假设参与者有道德自觉——用利害计算解释行为
3. 信息不足时不编造——明确说"以下判断依赖假设X，如果X不成立则结论不同"
4. 不是"知道答案"——是"知道怎么问问题"——框架给出结构，LLM填充域知识

## 质量门禁：三 Agent 管线（★v3）

> **构建者不能验证自己的输出。** 但仅此不够——如果 Challenger 读到初稿的完整推理过程，它会被带偏。从 shopping-claim-verify 验证的模式：**Challenger 只看到"数据层"（输出结论），不看到"推理层"（怎么得出这个结论的）。**
>
> v3 新增第三 Agent：**负空间检查**——框架本身也是盲区来源。追问"这个分析漏掉了什么视角？"

### 三 Agent 管线

```
用户问题
    │
    ├── Agent 1 (主分析, Sonnet/Opus)
    │   搜索域知识 → 五步框架分析 → 产出初稿
    │   输出切分为两层:
    │     - 数据层 (给 Challenger): 裂缝列表·合法伤害权归属·行动建议·博弈方
    │     - 推理层 (不给 Challenger): 为什么这样分析·利害计算过程·案例引用
    │
    ├── Agent 2 (信息不对称 Challenger, Haiku)
    │   ★ 只收到数据层——不看到推理层
    │   按四维攻击数据层:
    │   1. 裂缝遗漏 2. 合法伤害权归属错误
    │   3. 制度有效假设 4. 反位分析
    │   ↓
    │   发现盲区 → 标注修正 → 合并回初稿
    │   通过 → PASS
    │
    └── Agent 3 (负空间检查, Haiku)
        追问: "这份分析以什么为前提假设？"
              "哪个博弈方没有被当成独立视角来分析？"
              "如果换一个文化/制度背景，结论会变吗？"
        ↓
        输出: "以下视角未被覆盖: ..."
        追加到最终输出底部
```

### 信息不对称分离（★P0）

主分析 Agent 的输出必须分为两层——Challenger 只收到数据层：

| 层 | 内容 | 给 Challenger? |
|----|------|---------------|
| **数据层** | 裂缝列表（每条裂缝一句话）、合法伤害权归属（谁·什么裁量权·伤害/恩惠谁）、博弈方表（各方约束+利害计算）、行动建议列表（每条建议+代价） | ✅ 收到 |
| **推理层** | 为什么这样识别裂缝、利害计算的推导过程、案例引用、叙事性分析 | ❌ 不收到 |

**为什么：** shopping-claim-verify 的实测——Challenger 读到父 Agent 的完整推理后，确认偏误率显著升高（"它已经分析得很全面了"）。只给数据层，Challenger 被迫独立判断"这些裂缝够不够、这个人真的是唯一合法伤害权持有者吗"——不依赖父 Agent 的论证。

### Challenger 指令

```
你是潜规则分析的独立审查者（Challenger）。
你收到的是一份分析的"数据层"——只有结论，没有推理过程。
你的任务是判断：仅凭这些结论，分析是否有盲区。

四维攻击（每维一句话结论）：

1. 裂缝遗漏：列出的制度裂缝是否足够？
   ——用你的独立知识：还有谁有自由裁量权但没被列出？
   ——有没有信息不对称节点被遗漏？

2. 合法伤害权归属：裁量权的真正持有者是谁？
   ——列出来的人真的是唯一有伤害能力的人吗？
   ——有没有"二次分配"（权力给A，A转给B）？

3. 制度有效假设：行动建议中哪一条暗中假设了"制度会响应"？
   ——有没有"你去投诉/申诉/走流程"类的建议？
   ——有没有把"理论上可以"当成了"实际上会"？

4. 反位分析：如果被分析的掌权者读到这份结论，ta会怎么反驳？
   ——结论有把掌权者的行为归因于恶意/道德吗？（应该用利害计算解释）

输出格式：
- 如果无明显盲区：回复 "PASS"
- 如果发现盲区：指出具体缺失 + 一句话修正
- 不确定时标注 "LOW CONFIDENCE: ..."
```

### 审查标准

| 结果 | 含义 | 处理 |
|------|------|------|
| **PASS** | 无关键盲区 | 初稿直接输出，底部标注"✅ Challenger 已审查" |
| **MINOR** | 有次要遗漏 | 在输出中追加"💡 Challenger 补充"段落 |
| **MAJOR** | 关键盲区或错误 | 修正初稿对应部分，底部标注修正内容 |

## 输出许可等级（★P1）

> 从 cognitive-license 技能复用的模式。不是验证真假——是判每条建议能安全地用来做什么。

每条行动建议末尾标注许可等级：

| 等级 | 含义 | 用户可以做什么 |
|------|------|--------------|
| 🟢 **FOUNDATION** | 有多源证据支撑，制度裂缝可验证 | 据此做决策、投入资源 |
| 🟡 **CLUE** | 基于模式推断，有案例支撑但非确定性 | 作为判断线索，结合自己的信息验证 |
| 🟠 **FRAMEWORK** | 框架推理产物，逻辑自洽但域知识依赖搜索质量 | 用于理解结构，不作为唯一决策依据 |
| 🔴 **SPECULATION** | 信息不足下的推测 | 仅作为可能性之一，不作为行动依据 |

**分级标准**：
- FOUNDATION：正式规则可查证 + 裂缝有公开案例 + 合法伤害权归属有法律/合同依据
- CLUE：裂缝推断自结构逻辑 + 有类似域的案例支撑
- FRAMEWORK：纯框架推理（"给定这个制度结构，X类裂缝应该存在"）但未经本域搜索确认
- SPECULATION：信息不足以完成框架分析，标注"假设X为真则Y"

## 负空间检查（★P2）

> 从 decision-analyzer 复用的模式。框架本身是盲区来源——它让你看到某些东西，也让你看不到另一些东西。

在 Challenger 审查通过后，追加一步：

```
追问三个问题：
1. 前提假设：这份分析以什么为不言自明的前提？
   ——"业主大会有权决策"的前提是"业主大会存在"——如果不存在，整条推理链断裂
   ——把每个分析步骤的前提假设显式写出来

2. 缺失视角：哪个博弈方没有被当成独立视角来分析？
   ——初稿分析了物业、业主、街道办——但开发商呢？租户呢？
   ——列出"未被覆盖的博弈方"清单

3. 域边界：如果把这个情境放到完全不同的制度背景下，结论会变吗？
   ——同样的潜规则结构在什么条件下不成立？
   ——这暴露了分析中哪些隐含的制度/文化假设？
```

输出追加到最终分析末尾：
```
### 🔮 负空间检查

前提假设: [分析依赖的关键假设列表]
未被覆盖的视角: [博弈方/利益相关方]
域边界: [本分析在什么条件下不成立]
```

## 输出格式（★v3.1 人读优化）

**三 Agent 管线内部运行——最终输出是合成后的易读版本。** Challenger 的修正和负空间发现的盲区已合并到对应段落中，不单独追加。

```
## 一句话

[问题核心——不是复述问题，是点出制度裂缝的本质。≤30 字]

## 裂缝

| 裂缝 | 谁有裁量权 | 怎么伤害你 |
|------|-----------|-----------|
| ... | ... | ... |

## 权力格局

[一段话：合法伤害权在哪几方手里、谁跟你利益一致、谁跟你利益冲突]

## 行动建议

1. **[策略名]** [🟢FOUNDATION/🟡CLUE/🟠FRAMEWORK/🔴SPECULATION]
   做什么、代价、止损条件。≤3 行。

2. ...

**最低有效剂量**: [一句话]

## ⚠️ 我们可能漏了什么

[负空间——3 条前提假设 + 2 个未覆盖视角。≤5 行，极简]
```

### 格式铁律

- **不用学术术语**。说"裁量权"不说"合法伤害权"——概念留在分析过程，不在输出里炫技
- **每段 ≤5 行**。超过就拆
- **行动建议 ≤3 条**。只列值得做的，不列"理论上可以但实际上不行"的
- **"我们可能漏了什么"是必选项**——不写就说明没跑负空间检查
- Challenger 的修正**合并到对应段落**，不要单独追加"Challenger 说..."——读的人不关心管线结构

## 持久化（★落盘）

**每次分析完成后，将最终输出（经 Challenger 修正 + 负空间检查后的完整版）保存到磁盘。**

```
路径: workspace/unwritten-analyses/YYYY-MM-DD-{domain}-{slug}.md
      ↑ 相对于 Mino 工作区根目录

命名规则:
  - YYYY-MM-DD = 分析日期
  - domain = 职场/饭局/商业/平台/生活/法律/物业/其他
  - slug = 问题简述（取前 20 字，去标点，空格换-）

示例:
  workspace/unwritten-analyses/2026-07-19-物业-小区停车费收费争议.md
  workspace/unwritten-analyses/2026-07-19-平台-抖音流量骤降判断限流.md
```

保存内容 = 最终合成版（主分析 + Challenger 修正 + 负空间检查），Markdown 格式。
文件头包含元数据：日期、域、原始问题、许可等级汇总。

**为什么落盘**：分析投入了 token 和时间——不能会话结束就丢。落盘后可以回顾、对比、发现自己的判断盲区模式。

## 限制

- 框架是通用的，但域知识依赖实时搜索质量——搜索不到的信息会影响判断精度
- 不替代法律/合规/专业咨询——输出是分析框架，不是正式建议
- 框架的力量在"问对问题"——如果用户提供的情境描述不完整，分析质量受限于信息的完整度
