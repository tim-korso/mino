# task-alignment — 需求澄清

指令缺信息时，先对齐再动手。猜了返工的代价比多问一句大 10 倍。

## 三条硬触发

出现任一，立即反问：

1. **路径不明确** — 「看看代码」「改改那个文件」→ 哪个文件？什么路径？
2. **范围不明确** — 「优化一下」→ 优化什么？性能？结构？可读性？
3. **验收标准不明确** — 「搞好了就行」→ 什么算好？怎么验证？

## 澄清方式

`AskUserQuestion` + 2-3 个具体选项，每条带清晰描述。不给选项让用户自由发挥 = 没对齐。

```
AskUserQuestion({
  questions: [{
    header: "优化范围",
    question: "你说的「优化」具体指哪方面？",
    options: [
      { label: "性能", description: "减少 re-render，优化列表" },
      { label: "代码结构", description: "拆组件，提 hooks" },
      { label: "可读性", description: "注释、命名、清死代码" },
    ]
  }]
})
```

## 违规

- 模糊时猜了就干 → 违规
- 反问不给选项 → 违规
- 信息缺口影响结果正确性还继续 → 违规

## 与 Think 通道的分工

| 阶段 | 走什么 |
|------|--------|
| 需求模糊 | task-alignment → 反问澄清 |
| 需求明确但方案分叉 | Think 通道 `Agent()`（继承会话旗舰——独立上下文即 Think 价值；`model:"opus"` 实测降为 k2.6，勿用） |
| 需求明确路径唯一 | 直接执行（Fast） |
