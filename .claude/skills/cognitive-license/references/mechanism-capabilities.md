# Mechanism Capabilities — 模型能力边界参考

> 不是测评"这个模型强不强"。是记录**"这个机制在结构上能做什么、不能做什么"**。
>
> 机制 ≠ 模型。扩散是所有扩散模型的机制。自回归是所有 LLM 的机制。
> 模型可以变强，但机制的边界不因模型变强而消失——因为边界是数学的，不是工程改进能跨越的。

## 核心原则

**每个机制的产出方式决定了它的控制粒度和能力边界。产出方式不可改变。**

把任务分配给一个机制 = 把任务限制在这个机制的能力边界内。
要求一个机制做它边界外的事 = 结构性失败，不是"调参能解决"的。

## 1. 扩散模型（Diffusion）

### 机制
从噪声中逐步去噪生成图像。每一步对整个画面做全局调整，不存在"画这个元素"的操作——存在的是"让整个画面在这个区域逐渐呈现某种图案"。

### 控制粒度
**画布级**。能控制：整体风格、构图方向、色彩氛围、主体大致位置。
**不能控制**：单个元素的精确位置、元素的精确数量、文字/字符的准确形状。

### 结构性 CAN do
- 生成具有特定风格/氛围的图像
- 在 prompt 中描述的元素大概率会出现（但不保证精确位置）
- 生成图像变体（img2img、inpainting）
- 整体构图控制（三分法、对称、引导线等方向性指导）

### 结构性 CANNOT do
- **精确定位**：不能在指定坐标画一个指定大小的圆
- **文字渲染**：生成的"文字"是视觉图案模拟，不是可读文字——形状接近但字符不正确
- **精确计数**："画 7 个苹果"→ 可能画 5-9 个
- **保持一致性**：同一个角色/物体在多次生成中外观不可控地变化
- **增量编辑**：不能"把左边的手往右移 10px"——只能重新生成整张图

### 常见越界模式

| 越界请求 | 为什么不可能 | 被越界的机制边界 |
|---------|-------------|----------------|
| "在图片上标注 A、B、C 三个点" | 扩散生成的是视觉图案，标注位置无法精确控制 | 精确定位 |
| "生成一张图，上面写'欢迎光临'四个字" | 文字是图案模拟，四个字可能变成形状接近的图案 | 文字渲染 |
| "把这张图的左手改小一点" | 扩散只能整体重新生成，不能增量编辑单个元素 | 增量编辑 |
| "生成 5 个完全相同风格的角色" | 每次生成独立，没有跨生成的"角色一致性"机制 | 一致性保持 |

### 正确拆分
```
❌ 一个模型全包：扩散模型生成图片 + 标注文字
✅ 分而治之：
    ├── 扩散模型：生成图片（不做任何标注要求）
    ├── VLM：看图 → 定位语义区域 → 返回坐标
    └── SVG/Canvas：在坐标上叠加标注文字
```

### 实例
book-figure 项目：通义万相生成线稿 + Qwen-VL 定位部位 + SVG DPT-CP1 标注。
扩散做生成、VLM 做定位、SVG 做标注——三个机制各做各的结构性能力范围内的事。

## 2. 自回归语言模型（LLM）

### 机制
给定前缀，预测下一个 token。每一步只有"选哪个 token"这一个操作。输出是 token 序列。

### 控制粒度
**Token 级**。能控制：文本内容、风格、结构、逻辑链。
**不能控制**：执行——LLM 不是执行引擎，它产出的是文本，不是操作。

### 结构性 CAN do
- 文本生成：分析、总结、翻译、改写、创意写作
- 模式匹配：在给定信息中提取、分类、对比
- 推理：在 token 序列上做逻辑推演（前提：推理所需信息全在上下文里）
- 工具调用：生成工具调用的文本格式（但不执行——执行是外部机制的事）

### 结构性 CANNOT do
- **执行操作**：不能运行命令、调用 API、修改文件系统。它只能生成"运行这个命令"的文本
- **自我验证**：不能跳出自己的概率路径评估自己的输出质量（见 cognitive-license §冷启动原理）
- **内省**：不能观察自己的中间状态——输出是 argmax/sample，不是 introspection
- **知道自己不知道**：不能可靠判断"这个问题我的知识范围内没有答案"
- **精确计数/计算**：token 预测不经过算术逻辑——大数运算、精确统计需要外部工具
- **确定性操作**：相同 prompt 可能得到不同输出（取决于采样策略和随机种子）

### 常见越界模式

| 越界请求 | 为什么不可能 | 被越界的机制边界 |
|---------|-------------|----------------|
| "你帮我跑一下这个脚本" | LLM 产出文本，不是 shell 进程 | 执行 |
| "检查一下你刚才的分析有没有问题" | 同一条概率路径打分 = 系统性偏高 | 自我验证 |
| "你确定吗？" | LLM 没有"确定度"的内省访问——它可能输出"我确定"但实际不确定 | 内省 |
| "列出所有可能的风险" | 穷举不在 token 预测机制内——它能列出常见的，不能保证穷举 | 精确穷举 |
| "把这段代码里的所有 bug 都找出来" | 同上——能找常见的，不能保证全覆盖 | 精确穷举 |

### 正确拆分
```
❌ 一个 LLM 全包：生成方案 + 自我验证 + 执行
✅ 三角分离：
    ├── LLM-A：生成方案（Scout）
    ├── LLM-B：冷启动验证方案（Challenger，独立上下文）
    └── 外部执行器：运行验证步骤（Bash/API/Playwright）
```

## 3. 视觉语言模型（VLM）

### 机制
接收图像 + 文本 → 输出文本。模型在图像特征和文本 token 的联合空间里做预测。看图的方式 = 把图像切成 patches → 编码为特征向量 → 和文本 token 一起做注意力计算。

### 控制粒度
**语义区域级**。能控制：描述图中的语义内容、识别物体类别、定位"大概在哪个区域"。
**不能控制**：像素级精确坐标、精确文字识别、微小细节。

### 结构性 CAN do
- 语义识别：这是什么物体、什么场景、什么动作
- 区域定位：给出物体/部位的近似位置（"在图像的左下方"）
- 关系描述：A 在 B 的旁边/上面/里面
- OCR 粗读：对清晰、标准字体的文字可识别
- 风格/氛围判断：整体色调、风格、情感倾向

### 结构性 CANNOT do
- **像素级精确定位**：VLM 看图像的方式是 patches，不是像素。坐标是近似的，误差通常在 10-50px
- **文字精确识别**：对艺术字体、手写体、小字号、低对比度文字——VLM 的误读率系统性地高
- **细节识别**：微小的、低对比度的、被遮挡的物体——VLM 可能漏掉或误判
- **精确计数**：密集排列的小物体——VLM 的计数是估计，不是枚举
- **区分视觉相似但语义不同的图案**：扩散模型生成的轮廓碎片可能看起来像字母——VLM 可能把它们当真字母
- **跨图一致性**：两张图里的"同一个东西"可能被识别为不同的东西

### 常见越界模式

| 越界请求 | 为什么不可能 | 被越界的机制边界 |
|---------|-------------|----------------|
| "给出图中每个圆的精确(x,y)坐标" | VLM 的 patches 不保留像素级空间精度 | 像素定位 |
| "翻译图中所有文字" | 手写/艺术字/小字 → 系统性漏读和误读 | 文字精确识别 |
| "数一下图中有多少个点" | 密集排列 → VLM 估计，不是枚举 | 精确计数 |
| "判断这些字母碎片哪些是真正的字母" | 轮廓碎片和真字母在 patches 层面可能是相同特征 | 语义区分 |

### 正确拆分
```
❌ VLM 全包：识别 + 像素定位 + 文字提取
✅ 分而治之：
    ├── VLM：语义识别 → "左下角有一个手部"
    ├── CV 算法（HoughCircles/Haar Cascade）：精确坐标检测
    └── OCR 专用模型（PaddleOCR/Tesseract）：文字提取
```

## 4. 检索系统（Search / Vector DB）

### 机制
给定查询 → 在文档/向量空间中找相似/相关项 → 返回排序结果。排名基于相似度/相关性分数，不是"正确性"。

### 结构性 CAN do
- 找到与查询语义相关的文档
- 按相关性排序
- 返回大量候选供进一步筛选

### 结构性 CANNOT do
- **验证信息正确性**：检索返回的是"相似的内容"，不是"正确的内容"
- **判断信息完整性**：返回的 top-K 不一定覆盖所有相关信息
- **综合跨源信息**：检索返回碎片，不负责综合

### 常见越界模式
"搜一下 XX 的真实数据" → 搜索结果排名 ≠ 准确性。排第一的可能是 SEO 优化的错误信息。

### 正确拆分
```
❌ 搜索 → 直接引用第一个结果
✅ 搜索召回 → LLM 交叉验证 → 多源确认 → 置信度标注
```

## 5. 规则引擎（Hooks / Lint / Firewall / Gate）

### 机制
输入 → 匹配预定义规则 → 触发行动（阻断/放行/标记）。规则是确定的、预定义的。

### 结构性 CAN do
- 精确匹配已知模式（"如果文件路径包含 /etc/ → 阻断"）
- 硬性阻断：匹配 → 100% 阻断，不匹配 → 100% 放行

### 结构性 CANNOT do
- **理解上下文**：不能判断"这次操作虽然匹配了规则但实际上无害"
- **处理未知模式**：不能发现规则未覆盖的新漏洞
- **概率判断**：不能输出"这个操作有 70% 的概率是恶意的"

### 常见越界模式
用规则引擎做内容审核——规则挡不住语义层面的规避（同义词、委婉语）。

### 正确拆分
```
❌ 规则引擎 → 直接判定内容安全
✅ 规则引擎（精确匹配 + 硬阻断）→ 可疑内容 → LLM（语义判断）→ 最终决策
```

## 6. 变换引擎（ffmpeg / pandoc / ImageMagick）

### 机制
输入格式 A → 确定性变换 → 输出格式 B。变换规则是预定义的数学/算法操作。

### 结构性 CAN do
- 格式转换：保持内容不变，改变编码/封装
- 参数化处理：缩放、裁剪、码率调整

### 结构性 CANNOT do
- **理解内容语义**：pandoc 转换 markdown→docx 不理解文章在说什么
- **内容质量判断**：ffmpeg 转码不知道视频"好不好看"

## 7. 混合机制的边界：何时合并、何时拆分

### 合并规则
当两个零件需要同一个机制的能力，且合并不会让其中一个做它机制外的事 → 合并。

```
✅ 合并：VLM 看图 + 描述 = 同一个视觉理解机制，不越界
✅ 合并：LLM 分析 + LLM 总结 = 同一个文本生成机制，不越界
```

### 拆分规则
当任务包含两个不同机制的能力要求 → 必须拆分。不拆分 = 逼一个机制做它结构上不能做的事。

```
❌ 合并：扩散模型生成图片 + 在图上标注 = 两个不同机制（扩散+定位）
✅ 拆分：扩散（生成）+ VLM（定位）+ SVG（标注）

❌ 合并：LLM 生成方案 + LLM 自我验证 = 同一个机制但被要求做它不能做的事（自我验证）
✅ 拆分：LLM-A（生成）+ LLM-B 冷启动（验证）

❌ 合并：搜索找到数据 + LLM 判断数据质量 = 可以合并（LLM 判断是它机制内的事）
   但搜索+直接引用不加 LLM 判断 = 错误——检索排名≠准确性
```

### 拆分决策树

```
这个任务需要什么能力？
    │
    ├── 所有能力属于同一个机制？
    │     ├── 是 → 可以合并
    │     └── 否 → 必须拆分
    │
    └── 对每个子任务：分配给机制匹配的工具
        ├── 生成类 → 扩散 / LLM
        ├── 识别/定位类 → VLM / CV / OCR
        ├── 验证类 → 独立 LLM（冷启动）/ 规则引擎
        ├── 执行类 → Shell / API / Playwright
        ├── 检索类 → Search / Vector DB
        ├── 变换类 → ffmpeg / pandoc
        └── 硬阻断类 → Hook / Lint / Gate
```

## 维护

- 新模型出现时，判断它是新机制还是现有机制的变体
- 新机制 → 新增条目，记录结构性的 CAN/CANNOT
- 变体（同一机制，更强/更快）→ 不新增条目——机制边界不变
- 每次发现新的越界模式 → 更新对应机制的"常见越界模式"表
- 每次成功拆分修复了一个越界 → 更新对应机制的"正确拆分"示例

## 与 cognitive-license 集成

分级者在评估一个方案/计划时，使用本文档回答：
1. 每个步骤需要什么模型/工具？
2. 该模型/工具的产出机制是什么？
3. 这个步骤的要求是否落在该机制的能力范围内？
4. 如果越界 → 触发 `capability_boundary_violation` → 建议拆分方案
