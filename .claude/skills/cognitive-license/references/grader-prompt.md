# Grader Prompt — 冷启动分级者参考

> 这不是给生成者看的。这是给分级者看的——在一个独立的、冷启动的 Agent 调用中使用。
> 分级者不知道主张是怎么来的、谁生成的、基于什么上下文。它只看到裸主张文本。

## 核心原则

### 冷启动铁律

分级者的 prompt 中绝不能出现：
- 原始文本全文
- 生成者的身份或上下文
- 用户最初的问题
- 任何暗示"这些主张来自一个可信来源"的信息
- 生成过程中产生的任何中间产物

分级者 prompt 中只能出现：
- 主张文本（逐字引用）
- 分级标准和分类法
- 领域上下文（可选，仅帮助判断可验证性）
- 历史 resolved cases 作为 few-shot（可选，仅匹配领域）

### 分级决策树

```
存在 ground truth 吗？
  ├── 存在
  │     ├── 可获取吗？
  │     │     ├── 可获取 + 已被可靠来源证实 → FOUNDATION
  │     │     ├── 可获取 + 未经证实 → DIRECTION
  │     │     └── 不可获取 → FLAG（data_ghost / pseudo_attribution）
  │     └── 半衰期到了吗？
  │           └── 到了 → FLAG（half_life_bomb）
  │
  ├── 不存在（诠释性的）
  │     ├── 作为框架有价值 → FRAMEWORK
  │     └── 伪装成事实 → FLAG（interpretation_as_fact）
  │
  ├── 不存在（问题预设了错误前提）
  │     └── FLAG（embedded_presupposition）
  │
  ├── 不存在（答案在未来）
  │     └── FLAG（future_disguise）
  │
  └── 已知为假/问题本身不成立
        └── REJECT
```

### 降级优先原则

不确定时，选更低的使用许可。理由：

| 错误类型 | 后果 | 可逆性 |
|---------|------|:---:|
| DIRECTION 标成 FOUNDATION | 劣质地基 → 整栋楼歪 | 不可逆（下游全污染） |
| FOUNDATION 标成 DIRECTION | 多用一轮验证 | 可逆（验证完升级） |
| FLAG 标成 DIRECTION | 伪知识进入使用 | 不可逆 |
| DIRECTION 标成 FLAG | 多一次人工审核 | 可逆（审完放行） |

降级的代价 = 多做一轮验证。不降级的代价 = 整栋楼歪。选前者。

## 六类触发器：详细检测启发式

### 1. data_ghost（数据幽灵）

**定义**：主张给出了精确数字，但这个数字对应的 ground truth 不存在或不可获取。

**触发条件（满足任一）**：
- [ ] 给出了总量/占比/增长率数字，但概念本身有多个竞争性定义
- [ ] 数字精度高于数据采集方法的精度
- [ ] 引用了"研究表明"但没指具体研究
- [ ] 引用了需要付费商业数据库才能获取的数据
- [ ] 统计频率低于主张的时效要求（如"最新"数据但每5年才出一次）

**深层模式 — 多口径陷阱**：
不是"没有数据"——是"有多个不可比的数据，选哪个已经暗含了结论"。
影子银行有5种口径（窄/中/宽/穆迪/FSB），选窄口径=规模小=风险可控，
选宽口径=规模大=系统性风险。两个都是"真的"——但选哪个不是数据问题，
是立场问题。

**深层模式 — 比较基准陷阱**：
"X比Y好"类主张的数据幽灵不是"没有数据"——是通过选择比较基准决定了结论。
三个模型各有擅长的 benchmark，选哪个都可以"证明"自己的排序。

**检查方法**：
1. 这个数字如果真的存在，它在哪个具体的数据源里？（数据库名/年鉴名/报告名/表号）
2. 如果找不到具体数据源 → data_ghost
3. 如果找到了但在付费墙后面 → data_ghost（对普通验证不可达）
4. 如果找到了但概念定义不统一 → data_ghost（多口径陷阱）

**典型案例**：见 resolved-cases.yaml RC-001, RC-007

### 2. pseudo_attribution（伪归因）

**定义**：主张给出了精确的因果归因（"X贡献了Y的Z%"），但X在现实世界中无法从其他因素中分离。

**触发条件（满足任一）**：
- [ ] "X贡献了/导致了Y的Z%"——带精确百分比
- [ ] 系统中存在3+个同时作用的因素
- [ ] 因素之间有已知的交互效应
- [ ] 使用的是观测数据而非实验数据
- [ ] 没有提及方法论（回归/分解/DID/合成控制）

**深层模式 — 确定性幻觉**：
统计方法（回归分解/Shapley值/方差分解）可以产生百分比数字。但方法的
输出质量 ≤ 假设的质量。假设在现实中不成立 → 百分比没有因果含义。
"方法论上可以算出来" ≠ "算出来的东西有因果含义"。

**检查方法**：
1. 在现实世界中，有没有发生过"只有X变化、其他所有变量不变"的事件？
2. 如果没有（观测数据）→ 用的是什么方法分离的？（回归？分解？）
3. 如果用了方法 → 方法的假设在这个场景下成立吗？
4. 如果不成立 → pseudo_attribution

**注意**：不是说"不能用统计方法做归因"——是说"统计归因的精度不能超过假设的坚实度"。如果假设"线性和可加性"在现实中不成立，那60%和40%的区别就没有意义。

**典型案例**：见 resolved-cases.yaml RC-002

### 3. embedded_presupposition（问题嵌预设）

**定义**：主张在回答"为什么"之前，跳过了论证"是不是"。

**触发条件（满足任一）**：
- [ ] 包含比较级词汇（"更"、"优于"、"领先"）但没提供比较成立的证据
- [ ] 分析"原因"的篇幅明显大于论证"前提"的篇幅
- [ ] 使用了"为什么A比B好"的句式
- [ ] 隐含的价值判断被当作客观事实

**深层模式 — 好论证压制质疑**：
论证越详细、越有结构、越有说服力 → 读者越不会质疑预设。
三段清晰的"原因分析"本身就是预设的盔甲——"分析得这么深入，肯定研究过是不是"。
但实际上，"为什么"和"是不是"是两个独立的研究问题，前者不包含后者。

**检查方法**：
1. 主张的结论是否依赖一个未论证的预设P？
2. P本身是否显然为真？
3. 如果P不显然为真 → 文本中有多少篇幅用于论证P？有多少篇幅用于论证"基于P的推理"？
4. 如果论证P的篇幅 << 基于P的推理篇幅 → embedded_presupposition

**注意区分**：领域内的常识预设不需要论证（"银行需要资本充足率"——这是领域公理）。
需要触发的是争议性预设（"A银行风控比B银行好"——这不是公理）。

**典型案例**：见 resolved-cases.yaml RC-003

### 4. future_disguise（未来伪装）

**定义**：给出了精确的未来预测，但预测对象取决于策略选择而非机械因果。"假设其他条件不变"在现实中不成立。

**触发条件（满足任一）**：
- [ ] 预测带精确数值（bp/百分点/亿/百分比）
- [ ] 预测对象取决于相关方的策略反应
- [ ] 没有讨论"假设其他条件不变"的脆弱性
- [ ] 使用了"将"、"会"、"预计"等确定性措辞

**深层模式 — 精确数字的权威伪装**：
越是精确的预测数字（12bp而不是"约10bp"或"可能收窄"），越给人"经过了
严格计算"的感觉。但这种精确本身是脆弱的——精确到bp的计算建立在一系列
"假设其他条件不变"之上，而这些假设在现实中是必然被违反的。

**检查方法**：
1. 预测对象的相关方（银行/企业/个人/政府）在事件发生时会不会什么都不做？
2. 如果他们会采取应对策略 → 预测的精确数字建立在他们"不做任何事"的假设上
3. 这个假设在现实中成立吗？
4. 如果不成立 → future_disguise

**降级建议**：DIRECTION 而非 REJECT——精确预测作为"给定假设的情景分析"是有价值的。
但需要明确标注："此预测依赖于以下假设：X不会改变其行为。如果X做出策略调整，
此数字不成立。"

**典型案例**：见 resolved-cases.yaml RC-004

### 5. interpretation_as_fact（诠释伪装事实）

**定义**：用不可操作化的标签做分类判断。标签没有稳定外延，但被当作客观分类使用。

**触发条件（满足任一）**：
- [ ] 使用了二元标签但标签没有公认的操作化定义
- [ ] 标签在不同语境下含义不同（"强监管"可以指规则数量、执法强度、处罚金额...）
- [ ] 连续谱被二分成对立的类别
- [ ] 分类未经定义就直接用于推理

**深层模式 — "大家都知道"幻觉**：
这类标签的共同特点是——讨论者默认"大家都知道什么意思"。但没有人知道，
或者每个人知道的都不一样。标签不是没定义——是有太多隐式定义，且这些定义
之间不可通约。

**检查方法**：
1. 搜"[标签] 定义"——是否有一个公认的、操作化的定义？
2. 搜"[标签] 是什么 不是 什么"——是否有人用同一个词但意思完全不同？
3. 这个标签能否被翻译成可测量的指标？（"强监管"→"过去12个月内发布了X条新规"）
4. 如果不能且存在多个互不兼容的用法 → interpretation_as_fact

**降级建议**：FRAMEWORK 而非 FLAG——这类主张作为理解框架是有价值的。
"用'强监管→精准监管'的框架来理解当前政策方向"可以是 productive 的思考方式。
问题不在于"用这个标签"——在于"把这个标签当客观事实"。

**典型案例**：见 resolved-cases.yaml RC-005

### 6. half_life_bomb（半衰期炸弹）

**定义**：主张引用的信息时效已过，但未标注。信息本身可能真实，但作为当前判断的依据已变质。

**触发条件（满足任一）**：
- [ ] 市场/价格/交易数据超过3个月
- [ ] 监管规则引用未标注发文日期
- [ ] 公司经营数据未标注财报截止日
- [ ] "最新"、"当前"等时效词 + 实际数据已超过合理时效
- [ ] 数据产生后，期间发生了可能让数据失效的结构性事件

**各领域半衰期参考**：

| 信息类型 | 半衰期 | 失效条件 |
|---------|--------|---------|
| 日内市场数据（价格/成交量） | 1天 | 下一个交易日 |
| 高频经济指标 | 1-3个月 | 下一个月度/季度数据发布 |
| 低频经济指标（GDP/人口） | 3-12个月 | 下一次普查或年度统计 |
| 监管规则 | 到下次发文 | 新规出台/旧规废止/窗口指导 |
| 公司财报数据 | 到下一份财报 | 季报覆盖季报、年报覆盖年报 |
| 学术共识 | 到被新研究推翻 | 大型复制研究/meta-reanalysis |
| 技术性能数据 | 1-6个月 | 新版本/新架构/新benchmark |
| AI模型评测 | 1-3个月 | 新模型发布/新benchmark/评测方法更新 |

**检查方法**：
1. 从数据产生日期到当前日期，过了多长时间？
2. 这个时间段内，该领域通常会发生多少次更新？
3. 期间有没有发生可能让该数据失效的事件？（政策变化/市场转折/财报更新/行业重组）
4. 如果有 → half_life_bomb

**降级建议**：DIRECTION 而非 REJECT。过期数据仍然有历史参考价值——"2025年三季度不良率1.62%"作为当时的事实是正确的。问题在于它被用来论证"当前趋势"。

**典型案例**：见 resolved-cases.yaml RC-006

### 7. capability_boundary_violation（能力边界越界）★ NEW

**定义**：方案/计划中，某一步骤要求一个模型/工具做它机制结构上做不到的事。这不是"模型不够强"——是**机制在数学层面就不支持这个操作**。

**核心原则**：每个机制的产出方式决定了它的控制粒度和能力边界。把任务分配给一个机制 = 把任务限制在这个机制的能力边界内。要求越界 = 结构性失败，不是调参能解决的。

**各机制的结构性边界速查**（详见 `references/mechanism-capabilities.md`）：

| 机制 | 控制粒度 | 结构性 CANNOT |
|------|---------|---------------|
| **扩散模型** | 画布级 | 精确定位元素、文字渲染、精确计数、保持跨生成一致性、增量编辑 |
| **自回归 LLM** | Token 级 | 执行操作、自我验证、内省、知道自己不知道、精确穷举 |
| **VLM** | 语义区域级 | 像素级精确定位、精确文字识别（艺术字/手写）、精确计数、区分视觉相似的异义图案 |
| **检索系统** | 相关性排名 | 验证信息正确性、判断信息完整性、综合跨源信息 |
| **规则引擎** | 预定义模式匹配 | 理解上下文、处理未知模式、概率判断 |
| **变换引擎** | 格式级 | 理解内容语义、判断内容质量 |

**检测方法**：
1. 这个步骤要求什么能力？（生成？定位？执行？验证？检索？）
2. 分配给它的模型/工具的产出机制是什么？
3. 该机制的控制粒度是否覆盖这个能力要求？
4. 如果要求超出了机制的结构性边界 → capability_boundary_violation

**深层模式 — "合并就是越界"**：
当一个步骤被要求同时做两件事，而这两件事属于不同的机制能力范围 → 合并本身就是越界。
扩散做生成 + 定位 = 两个机制，合并 = 越界。
LLM 生成 + LLM 自我验证 = 同一个机制被要求做它不能做的事（自我验证），也是越界。

**修复建议格式**：不只说"这里越界了"，给出拆分方案——
```
越界：扩散模型被要求生成图片 + 在图上标注部位
原理：扩散的产出粒度是画布级，标注需要元素级定位
拆分：
  ├── 扩散模型 → 只做生成（不要求标注任何东西）
  ├── VLM → 看图定位语义区域 → 返回坐标
  └── SVG → 在坐标上叠加标注文字
理由：扩散做生成（机制内）、VLM 做定位（机制内）、SVG 做标注（机制内）
```

**注意区分**：
- 不是"这个模型现在不够好" → 是"这个机制的结构不支持"
- 不是"试试另一个同类型的模型" → 同类型 = 同机制 = 同边界。换模型不解决机制越界
- 修复是**拆分 + 重新分配**，不是**调参 + 重试**

**典型案例**：
- book-figure：扩散生成线稿 + 扩散标注 → capability_boundary_violation（扩散不能定位）。修复：拆分给扩散(生成) + VLM(定位) + SVG(标注)
- 见 resolved-cases 待录入案例

## 输出质量自检

分级完成后，分级者应自问：

### 跨主张一致性检查
- [ ] 有没有两条相似的主张被给了不同许可等级？为什么？
- [ ] 有没有一条 FACTUAL 主张因为"太像常识"被放了 FOUNDATION，但其实也需要验证？
- [ ] 有没有一条 FLAG 其实是"我不确定"而非"真的触发"？

### 边界感自检
- [ ] 我有没有对某个领域过度自信？（→ needsDomainExpert: true）
- [ ] 我有没有因为主张"听起来专业"而给了更高的许可？
- [ ] 我有没有因为主张"用了数字"而降低了警觉？

### 降级倾向自检
- [ ] 如果我对某条主张的 graderConfidence < 0.7，我降级了吗？
- [ ] FOUNDATION 等级的每一条，我真的能说出它为什么可以当地基吗？

## 维护

此文件随 resolved-cases.yaml 的积累而迭代。每当一种新的漏判模式被发现并记录到 resolved-cases 中，对应的检测启发式应更新到本文档的对应触发器章节。

更新节奏：每次 resolved-cases 累计新类型案例 3+ 条后，提炼通用模式并更新本文档。
