# /看清 — 工具机制四问

> 在碰工具之前，先问机制。
> 不是"怎么用"——是"它到底怎么运作的"。

## 何时触发

- 接触新工具/新模型/新 API/新协议
- 反复尝试"让 X 做 Y"但一直失败
- 设计方案前需要评估"谁适合干什么"
- 用户说"看清一下这个"

## 铁律

**每次"怎么让 X 做 Y"反复失败——不是 X 不够强，是 Y 不在 X 的机制范围内。**

不要试第三种变体。停下来，跑这个框架。

## 四问

### 问 1：机制——它怎么产出的？

不关心"它能做什么"，关心"它的输出是怎么来的"。

| 产出机制 | 例子 | 能控制什么 | 不能控制什么 |
|----------|------|-----------|------------|
| **扩散** | 图像生成模型 | 整体风格、构图方向 | 单个元素的精确位置 |
| **自回归** | LLM 文本生成 | 文本内容、结构、逻辑 | 执行——它不是执行引擎 |
| **检索** | 搜索引擎、向量 DB | 召回范围、排序 | 信息质量——召回≠准确 |
| **规则** | hooks、防火墙、lint | 硬性阻断/放行 | 上下文理解 |
| **变换** | pandoc、ffmpeg | 格式转换保真度 | 内容语义 |
| **查询** | SQL、GraphQL | 数据范围、聚合逻辑 | 数据本身的含义 |
| **事件驱动** | WebSocket、SSE、cron | 触发条件和时机 | 消费端的状态 |
| **缓存** | Redis、文件缓存、CDN | 命中率、失效策略 | 数据新鲜度——快≠新 |
| **流式** | 视频流、LLM streaming | 首字节延迟、背压 | 随机访问（流不是数组） |
| **同步/异步** | HTTP/RPC vs MQ/pub-sub | 响应时序 | 消费端的处理能力 |

**关键追问**：它的控制粒度在哪一层？画布级？元素级？token 级？字节级？

### 问 2：市场——还有谁能做这个？

不只问"谁最好"，问"谁的机制最匹配这个任务"。

```
备选方案对比模板：

| 方案 | 机制 | 强项 | 弱项 | 匹配度 |
|------|------|------|------|--------|
| A    | ...  | ...  | ...  | 高/中/低 |
| B    | ...  | ...  | ...  | 高/中/低 |
| C    | ...  | ...  | ...  | 高/中/低 |

→ 选 X：因为需要它的 [机制特征]，而 [弱项] 在这个任务中不致命
```

**跨域实例**：

```
任务：浏览器中执行用户代码 → 沙箱
  Web Worker  → 隔离线程，不能访问 DOM——适合纯计算
  iframe      → 隔离 DOM 但同源策略复杂——适合渲染隔离
  WASM        → 接近原生性能，无浏览器 API——适合计算密集型
  Playwright  → 完整浏览器环境，慢但真实——适合 E2E 测试
  
任务：Agent 间通信
  stdio/NDJSON → 进程间，最简单，无网络开销
  HTTP/REST   → 跨机器，同步请求-响应
  WebSocket   → 持久连接，双向推送
  MCP         → 工具调用标准化，带协议协商
  message queue → 完全解耦，异步消费
```

### 问 3：拆分——这个问题的零件是什么？

```
具体问题：
    │
    ├── 零件 A（需要 ___ 能力）→ 对照问1+2，谁做？
    ├── 零件 B（需要 ___ 能力）→ 谁做？
    └── 零件 C（需要 ___ 能力）→ 谁做？

⚠️ 自检：有没有两个零件合并给一个工具？
→ 合并 = 逼一个工具做它机制外的事 → 预计失败
```

**实例——不只 AI**：

```
问题：做一个实时的 Agent 对话仪表盘
  表面直觉：一个 WebSocket 全搞定 → ❌ 推送和渲染合并了
  正确拆分：
    ├── 状态变更推送 ── WebSocket（事件驱动，它只会这个）
    ├── 历史数据查询 ── REST API（请求-响应，适合按需拉取）
    ├── UI 渲染 ────── React 组件（状态→DOM）
    └── 数据持久化 ──── SQLite（结构化查询）

问题：批量处理 1000 个文件
  表面直觉：一个 for 循环 → ❌ 没考虑容错和并行
  正确拆分：
    ├── 文件发现 ──── glob（文件系统遍历）
    ├── 并行处理 ──── 线程/进程池（CPU 密集用进程，IO 密集用线程）
    ├── 错误收集 ──── 独立错误 channel（不全军覆没）
    └── 结果聚合 ──── reduce（拼回整体）
```

### 问 4：接口——零件之间怎么接？

相邻零件的输出格式必须等于下一个零件的输入格式。

如果 A 的输出不是 B 的输入 → 中间需要适配层。适配层本身是一个新零件。

```
A 的输出(格式) → [适配?] → B 的输入(格式)
B 的输出(格式) → [适配?] → C 的输入(格式)
```

**适配是有成本的**——如果适配层太厚，可能说明问 3 的拆分不对。厚的适配 = 重新思考。

**经典实例**：Unix pipe——`cat file | grep pattern | sort | uniq -c`。每个程序只做一件事。接口是 text stream。换掉任何一个（`grep`→`rg`、`sort`→`gsort`）其他不动。这就是问 4 的理想状态。

## 反模式

| # | 反模式 | 症状 | 根因 |
|---|--------|------|------|
| 1 | **合并幻觉** | "让 X 顺便做 Y 不就行了" | 跳过问 3——X 的机制不覆盖 Y |
| 2 | **金锤子** (Maslow's Hammer) | "我最熟 X，什么问题都用 X 解决" | 跳过问 2——认知偏误：熟悉≠匹配。Abraham Maslow 1966："如果你只有一把锤子，所有东西看起来都像钉子" |
| 3 | **功能蔓延** | "X 也能做 Y，虽然不是设计来干这个的" | 跳过问 1——能用≠机制匹配 |
| 4 | **适配过厚** | A→B→C 链路中写了大量转换代码 | 问 4 的反馈信号——回去重做问 3 |

## 思想渊源

这些原则不是我们发明的。这个 skill 把三条经典设计原则操作化成了可执行的四个问题：

| 原则 | 出处 | 映射到四问 |
|------|------|-----------|
| **Separation of Concerns** | Dijkstra 1974 | 问 3 拆分——每个 concern 一个零件 |
| **Do One Thing Well** | Unix Philosophy (McIlroy 1978) | 问 1 机制 + 问 3 拆分——一个工具只做它机制范围内的事 |
| **Rule of Composition** | Unix Philosophy — "Write programs to work together" | 问 4 接口——输出格式 = 输入格式，pipe 连接 |
| **Rule of Separation** | Unix Philosophy — "Separate policy from mechanism" | 问 1 机制——政策可变，机制固定；不要绑一起 |
| **Gall's Law** | John Gall 1975 — 能跑的复杂系统都是从能跑的简单系统进化来的 | 元层——先建能跑的最小拆分，再迭代加零件。不要一上来设计完美架构 |
| **Law of the Instrument** | Kaplan 1964 / Maslow 1966 — 过度依赖熟悉工具 | 反模式 2——熟悉工具是舒适的陷阱 |

## 输出格式

```markdown
## 看清：<问题/工具名>

### 机制矩阵
| 组件 | 产出机制 | 能做 | 不能做 |
|------|---------|------|--------|

### 市场扫描
| 方案 | 机制 | 匹配度 | 选定理由 |
|------|------|--------|---------|

### 零件拆分
A（能力要求）→ 工具 X
B（能力要求）→ 工具 Y
C（能力要求）→ 工具 Z

### 接口链
X.输出(格式) → Y.输入
Y.输出(格式) → Z.输入

### 风险
- X 不能做 Y 的事（机制限制）
- 适配成本评估
```

## 完整示例：看清 Workflow 编排引擎

示例演示四问如何发现非显而易见的机制边界——即使是对一个已广泛使用的工具。

### 机制矩阵

| 组件 | 产出机制 | 能做 | 不能做 |
|------|---------|------|--------|
| agent() | 指令→spawn LLM Agent→返回 text/schema JSON | 并发控制、schema 结构验证 | 内容正确性验证、Agent 间通信 |
| pipeline() | items×stages 独立流 | 无 barrier 延迟 | 跨 item 聚合——每个 item 各走各的 |
| parallel() | 全屏障：等待所有 thunks | 收集全部结果做下一步 | 流水线——必须等最慢的 |
| schema 参数 | JSON Schema→工具层强制验证输出格式 | 保证 JSON 结构正确 | 保证内容正确——结构对≠语义对 |

### 市场扫描

| 方案 | 机制 | 匹配场景 | 不匹配 |
|------|------|---------|--------|
| Workflow | 确定性 JS + 并发约束 | 已知结构的并行/流水线 | 运行时才发现任务结构 |
| 手动 agent() | 对话中逐个 spawn | 2-3 个子任务 | 10+ agent、需 barrier |
| Goal Loop | 迭代直到标记完成 | 目标明确但路径未知 | 预先知道结构的并行 |
| Task Center | 任务队列 + 状态机 | 持久状态、长时间 | 实时多 Agent 协调 |

### 零件拆分——发现的四个边界越界

```
多 Agent 协作
    ├── 调度（顺序/依赖）── Workflow 核心 ✅
    ├── 验证（产出对吗？）── 当前：schema 只验结构 ❌ 误当内容验证
    └── 恢复（挂了怎么办？）── 当前：返回 null ❌ 丢失失败原因
```

| 发现 | 本质 |
|------|------|
| pipeline() 名不副实 | 名字暗示 Unix pipe（数据流过 stages），实际是 per-item stream |
| parallel() 被名字吸引滥用 | "parallel" 比 "pipeline" 更直观表达"我要并行"→ 过度使用 barrier |
| schema 验证被当成内容验证 | 合并幻觉——结构验证≠语义验证 |
| null 丢失失败语义 | 下游无法根据失败原因做分支 |

### 接口链

```
agent() 输出(text) → 手动拼 prompt → 下一个 agent() 输入
                          ↑
              适配过厚：每次重建上下文，丢失上游的排除项/置信度
```

### 改进方向：三层分离

```
当前：调度+验证+恢复 全塞进 agent()
    ↓
改进：
  调度层（Workflow 核心，不变）
  验证层（独立 agent，做否定性搜索验证内容）
  恢复层（结构化错误 + 条件重试而非 .filter(Boolean)）
```
