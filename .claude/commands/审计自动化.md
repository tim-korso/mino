# /审计自动化 — Quarterly Automation Audit

> unwritten 镜头在技术系统的首次跨域映射。不替代 mac-probe（测能力）和 04-MEMORY（记教训）——找它们共同指向但从未被说出来的模式。

## Pipeline

### Step 1: 加载基线

读三个源：
- `.claude/skills/macos-automation/SKILL.md` 的"已知限制"表 → 这就是已有认知的边界
- `memory/04-MEMORY.md` 中所有 Lessons Learned → 踩过的坑
- 运行 `mac-probe.sh all` → 当前能力真实状态

### Step 2: 跑 unwritten 四维分析

对以下四个维度逐一审视——**关键约束：只输出不在已知限制表里的发现**。如果一个发现已经在 SKILL.md 的"已知限制"或 04-MEMORY 里有记录，跳过。这不是复习——是找新东西。

**维度 1: 裂缝** — 哪些故障模式在跨工具复制？

```
不列单个工具的 bug。找同一类故障出现在三个以上工具的模式：
  - BSD/GNU 差异导致的静默错误——不仅是已知的 sed/grep/stat/ps。
    还有哪些工具我们一直用 GNU 思维在操作但没意识到差异？
  - iCloud CloudKit 覆盖——已知 Mail 规则。还影响哪些我们没注意的自动化目标？
  - macOS 小版本更新带来的 CLI 输出格式变更——已知 spctl/socketfilterfw/XProtect。
    还有哪些工具的输出我们正在解析但从未验证跨版本稳定性？
  - SwiftUI AX 黑箱——已知 System Settings/Mail 设置。
    还有哪些 App 悄悄切了 SwiftUI 而我们不知道？
```

**维度 2: 资源格局** — 130 个工具，实际在用多少？

```
对比 Stage 1-12 的工具声明和实际使用频率：
  - 哪些工具在 SKILL.md 里占了一整节但从没被任何管线引用？
  - 哪些工具我们每两天就手动用一次但 SKILL.md 没收录？
  - Homebrew 22 个工具——哪些升级后静默改变了行为而我们没注意到？
```

**维度 3: 天花板偏移** — App 自动化天花板矩阵还准吗？

```
原矩阵 (2026-07-18 实测):
  - 有哪些 App 我们标记了"可自动化"但再测可能已经变了？
  - 有没有新 App 我们已经在自动化了但没更新矩阵？
  - macOS 26.x 小版本更新有没有打开之前封死的路或关上之前开通的路？
```

**维度 4: 一句话诊断** — ≤30 字概括自动化生态的真实健康状态

### Step 3: 分类新发现

对每条不在已知限制中的发现，标注：

```
严重度: 🔴 blocker（自动化管线静默失效）/
        🟠 major（持续消耗但未察觉）/
        🟡 minor（潜在风险）
可验证性: 可以用什么命令/测试验证这条发现？
修复路径: 短期（一周内可做）/ 中期（下个季度前）/ 长期（需要生态变化）
```

### Step 4: 输出审计报告

```markdown
# 🕳️ macOS 自动化盲区审计 — YYYY-QN

## 一句话诊断
≤30 字

## 新发现裂缝
N 条不在已知限制表中的新发现

## 天花板偏移
矩阵需要更新的条目

## 资源格局
工具使用 vs 声明的差异

## 下季度优先修复
Top 3
```

### Step 5: 更新止损条件

- 如果本次审计新发现 ≤2 条 → 降级为按需触发。unwritten 框架对技术系统的适用性标记为"有限"
- 如果新发现 ≥5 条 → 验证通过。继续季度节奏
