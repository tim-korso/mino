# /巡田 — Weekly Field Inspection

> 守望者的每周仪式。打开仪表盘，巡每一块田，做判断。

## Pipeline

### Step 0: 预检（新增 ★）

**在读取任何文件之前，先回答三个问题：**

1. **我在哪台机器上？**
   - `uname -s` → macOS 或 Windows？
   - macOS：cron 注册表 + KB 数据库可用
   - Windows：cron 状态需 `myagents cron list` 验证，KB 数据库路径不存在

2. **搜索 API 还活着吗？**
   - 快速探针：Exa/Tavily/Serper 各跑一次最小搜索（1 个 query）
   - 可用引擎列表 → 决定后续扫描用什么
   - 若双耗尽 → 立即注册 WebSearch 为备用引擎
   - **不要到 Step 3 才发现 API 死了**

3. **上次扫描是什么时候？**
   - 读 `memory/fields/watchlist-scan-log.md` 最后一行的时间戳
   - staleness > 24h → ⚠️ 检测器断链告警
   - staleness > 48h → 🔴 严重断链，本次巡田最高优先级修复

### Step 1: 读仪表盘 + 验证上次决策

Read `memory/FARM.md`。

额外检查：
- 对比 FARM.md header 的"上次巡田"日期和巡田日志最后一条的日期——是否一致？
- 上次"需要决策"清单中标记为已完成的——是否真的完成了？（读对应文件验证）
- 上次"需要决策"中未完成的——本次优先处理还是继续延期？

### Step 2: 逐田巡查

对每块田（`memory/fields/*.yml`，跳过 `.template.yml`）：

1. 读 field YAML — 当前状态、上次收割、health notes
2. **休耕田跳过** — `status: fallow` 的田只登记，不深度检查
3. 检查：
   - 最后一次产出的日期 — staleness > harvest_trigger.staleness？
   - health 标记 — `detectors_connected: false` 需确认原因
   - `d5_end_to_end_tested: false` → 累计多少次巡田标记了？（≥3 = 告警疲劳）
   - harvest_trigger 条件是否满足？
4. 如果有对应 topic 文件，读 Quick Reference 部分（前 50-80 行）— 了解最近进展
5. **跨平台判断**：如果 health 中引用了 macOS 专属路径（如 `~/.myagents/kb/`）而当前在 Windows 上 → 标注"跨平台不可用"而非"管线失效"

### Step 3: 检查检测器信号

**3a. 金融监管田 (Field 1)**

- macOS：`sqlite3 ~/.myagents/kb/d5_monitoring.db "SELECT ..."` → WoW Diff
- Windows：该 DB 不存在。检查 topic 文件中最后一次 Flash 扫描的日期 → staleness 天数
- 如果 Flash cron 断更 > 7 天 → ⚠️

**3b. 科技公司田 (Field 2)**

读取顺序（从轻到重）：
1. `memory/fields/watchlist-scan-log.md` — 最新扫描记录（按公司分组，最新在前）。**只读最后 200 行**，不读全量
2. `memory/topics/cognitive-gap-watchlist.md` — **只读"本周扫描汇总"表格（最新一个）** + 推荐部分
3. 如果 scan log 最新条目 > 24h → 检测器断链。立即判断：API 配额问题还是 cron 故障？
4. 如果检测器断链 → **启动 WebSearch 手动补扫**（见 Step 3c）

**3c. WebSearch 补扫流程（检测器断链时）**

```
对每家公司并行 WebSearch：
  - 公司名 + 股票代码 + "2026年7月 最新"
  - 重点信号公司额外搜：股价/财报/产品发布

覆盖范围：
  - 全部注册公司（当前 7 家 + Kimi K3 候选）
  - 每条 WebSearch 的结果提取：修复进度变化 Δ、空白幅度变化 Δ、新触发信号

扫描结果写入：
  - memory/fields/watchlist-scan-log.md → 追加扫描 #N 条目
  - memory/topics/cognitive-gap-watchlist.md → 追加"本周扫描汇总"段落
```

### Step 3.5: 信号质量审计

**触发条件（精简——避免每个信号都审）：**

仅当以下条件**同时满足两条或以上**时才跑 cognitive-license：
- 修复进度变化 ≥ 5pp **且** 该变化来自非官方披露来源（不是财报/公告）
- 叙事突变标记触发 **且** 基于诠释性标签而非可观测事实
- 该信号将直接驱动一次 costly 行动（deep research / 建新田 / 资源重分配）

**不触发审计的情况：**
- 信号来自港交所/上交所公告 → 官方披露，跳过审计
- 信号来自多源交叉验证（≥3 个独立来源）→ 跳过审计
- 变化 ≤ 2pp → 太小，不值得审计

**审计执行：**
1. 提取信号摘要（~150 字）
2. 在巡田报告的分析段落中手动做 quick check（不跑完整 Workflow——太贵）：
   - 这个数字有可追溯的数据源吗？
   - "修复进度"是操作化指标还是主观判断？
   - 有没有反方向信号被忽略了？

**寒武纪案例教训（保持）**：组合修复-7%、股价-16.2%、"预告异常"标签——三条高强度信号叠加产生"必须 deep research"的判断。但每条信号本身的信息质量从未被独立审计。

### Step 4: 判留/弃/再种

对标记了 ready_for_harvest 或汤姆关注的产出：

- **留** — 有价值 → 更新对应 topic 文件，更新 field YAML `last_harvest`
- **弃** — 没价值 → 在 FARM.md 巡田发现中记原因
- **再种** — 方向对但不够 → 更新 field YAML 的 `seed.key_claims`，**同时在 FARM.md 决策清单中写具体下一步动作**（不能只写"再种"）

**休耕判定（新增 ★）：**
- 同一未完成项被 ≥3 次巡田标记 → ⚠️ 告警疲劳。升级处理——找根因或承认做不到
- 同一未完成项被 ≥4 次巡田标记 → 标记 fallow（`status: fallow`），下次巡田跳过
- fallow 超过 30 天 → 下次巡田考虑删除该田

**分派追踪（新增 ★）：**
- 上次巡田"已分派"的 Agent 任务 → 本次必须检查是否完成
- 如果未完成 → 不是再次分派，是找为什么没完成（API 死了？忘了？太贵？）
- "再分派"最多一次。两次分派未执行 → 放弃或换方案

### Step 5: 更新仪表盘 + 数据文件

更新以下文件（按顺序）：

1. **`memory/fields/watchlist-scan-log.md`** — 如果有新扫描数据，追加条目
2. **`memory/topics/cognitive-gap-watchlist.md`** — 追加"本周扫描汇总"段落（修复进度表+变化要点+推荐）
3. **各 field YAML** — 更新 `health.notes`、`last_harvest`（如有收割）
4. **`memory/FARM.md`**：
   - 更新 header "上次巡田"日期
   - 更新田块状态表（修复进度、健康标记、建议）
   - 写本周巡田发现（逐田详细分析）
   - 更新资源消耗估算
   - 更新需要决策的清单（标记上次决策的执行状态）
   - 追加巡田日志条目

**FARM.md 更新规则：**
- 田块状态表的"最新产出"列——摘要不超过 3 行。详细分析放"本周巡田发现"
- 决策清单中上次已完成项 → 保留但标记 `✅`，不删除（留下决策轨迹）
- 决策清单中上次未完成项 → 判断：继续推进还是放弃？放弃的写原因

### Step 5.5: 潜规则洞察

> 仪表盘告诉你每块田的显性状态。unwritten 镜头告诉你它们共同指向的系统性裂缝。

对农场状态跑 unwritten 分析——不是描述"各田健康分"，而是找跨田模式。

**裂缝** — 同一类问题出现在 ≥2 块田：
- 检测器断链是否同时出现在 Field 1 和 Field 2？同一个根因吗？
- cron success ≠ AI 产出——和 6/17 速递静默失败同源吗？
- 告警疲劳——"major"标记存活超过 3 周？
- "Agent 已分派≠已执行"出现了几次？横跨哪些田？

**资源格局** — 实际分配 vs 声明分配：
- priority: 1 的田实际 token 是否少于 priority: 4 的田？
- 哪块田吃了最多的巡田篇幅（注意力垄断）？
- 非田消耗（写书管线等）是否在资源表中有行？
- 跨田授粉规则写了但从未执行——是设计问题还是执行问题？

**仪表盘盲区** — 什么信号根本没被采集：
- 有对话中反复出现的需求但没建田？
- 幻觉事件是否触发了任何田的健康告警？
- 跨平台状态（macOS vs Windows）是否被建模？
- 搜索 API 配额是否在耗尽前有预警？

**一句话诊断** — ≤30 字概括农场真实状态。不是"5 块田 3 块活跃"（FARM.md 第一行说了）。是"两块核心田空转两周，检测器双耗尽是根因"。

输出方式：作为巡田报告的最后一个 section——`## 🕳️ 潜规则洞察`。

### Step 6: 输出巡田报告

**输出质量规则：**

1. **禁止裸标签** — "稳定""健康"必须有可观测指标。不能说"Flash cron 稳定"，要说"近 7 次执行成功率 7/7，最近异常 2026-07-10"
2. **所有数字标注来源** — `[实测]`/`[估算]`/`[引用:来源]`
3. **预测用非确定语气** — "据商汤官方公告，预计 7/18 WAIC 发布"，不写"明天发布"
4. **效率数字只报行数，不推断认知** — "骨架 1290 行，原 1785 行"，不写"阅读量减少 73%"

**报告格式：**

```
🌾 巡田报告 — YYYY-MM-DD

## 预检状态
- 平台：[macOS/Windows]
- 搜索 API：[可用引擎列表] | [耗尽引擎列表]
- 上次扫描：YYYY-MM-DD HH:MM（staleness X 天）

## 本周收割：N 块田
| 田 | 产出 | 判定 | 原因 |

## 田块健康
✅ N 块 | ⚠️ N 块 | ❌ N 块 | ⚪ N 块休耕/休眠

## 核心发现
（逐田深度分析——聚焦变化和趋势，不重复田块状态表）

## 🕳️ 潜规则洞察
（跨田裂缝 + 资源格局 + 仪表盘盲区 + 一句话诊断）

## 需要决策
（优先级排序：🔴 > 🟡 > ⚪）

## 上次决策执行情况
| 决策 | 状态 | 结果 |
（追踪上次巡田的决策是否被执行）

下次巡田：YYYY-MM-DD
```

**数据展示与关键对比（★ 核心输出能力）：**

> 巡田不是数据罗列——是对比驱动的决策辅助。每个 widget 回答一个具体的对比问题。

### 对比维度矩阵

巡田的核心信息可以通过以下对比维度组织。每次巡田从矩阵中选取 ≥3 个维度输出 widget：

| 对比维度 | 回答的问题 | 最佳图表 | 何时必须用 |
|---------|-----------|---------|-----------|
| **时序对比** | "比上周好了还是差了？" | 分组柱状图（旧 vs 新） | 每次巡田 |
| **截面排名** | "谁最好谁最差？" | 水平条形图（排序） | 公司 ≥5 家时 |
| **速度对比** | "谁在加速谁在减速？" | 瀑布图（Δ 值） | Δ 绝对值 ≥3pp 的公司 ≥2 家 |
| **裂口对比** | "认知滞后 vs 实际进展？" | 散点图（修复 vs 空白） | 每次巡田 |
| **信号密度** | "谁的催化事件最多？" | 热力矩阵（公司×信号类型） | 新触发 ≥3 条 |
| **资源分配** | "谁在吃资源谁在挨饿？" | 堆叠柱状图（预算 vs 实际） | 非田消耗 > 田内消耗时 |
| **风险矩阵** | "谁最危险？" | 象限图（修复 vs 空白） | 有公司空白扩大或修复倒退 |
| **决策追踪** | "上次说的做了吗？" | 状态表（决策→执行→结果） | 每次巡田 |

### Widget 模板库

**模板 A：修复进度时序对比（每次巡田必出）**

双色分组柱状图。旧值灰调、新值彩色（涨=Teal、跌=Coral、平=Stone、新=Sky）。柱顶标注 Δ 值。

关键洞察提取：
- 最大涨幅公司 → 什么驱动的？（新信号/催化兑现/市场修复）
- 最大跌幅公司 → 根因是什么？（失信/负面数据/结构问题）
- 如果涨跌相抵后组合加权变化 <2pp → "表面平静但内部剧烈重分配"

**模板 B：∆ 瀑布图（变化剧烈时必出）**

横向瀑布图，从上次组合修复值出发，每家公司的 Δ 依次叠加，落到本次值。绿涨红跌灰平。

关键洞察提取：
- 正向驱动总和 vs 负向驱动总和 → 谁主导？
- 单家公司 Δ 超过总 Δ 的 50% → "这家公司独自扛着组合走"
- 多空双方都 ≥3pp → "两极分化加剧"

**模板 C：修复 vs 空白 散点图（每次巡田必出）**

X 轴=修复进度（越高越好），Y 轴=空白幅度（越低越好）。四象限：
- 右上（高修复+低空白）=🥇 成熟区（百济）
- 左上（低修复+低空白）=🥈 信息对称区（DS）
- 右下（高修复+高空白）=⚠️ 认知滞后区（智谱）
- 左下（低修复+高空白）=🔴 双低风险区（Kimi、优必选）

每个公司是一个圆点，大小=本周信号密度（新触发数+Δ 绝对值）。箭头指示本周移动方向。

**模板 D：信号触发热力矩阵（新触发 ≥3 条时）**

SVG 表格。行=公司，列=信号类型（技术/产品/商业化/融资/监管）。颜色深浅=信号强度。已触发=实心、接近触发=条纹、远期=灰色。

关键洞察提取：
- 哪列最密集 → 本周的隐性主线（如"算力基础设施"同时出现在 DS/智谱/商汤）
- 哪行最稀疏 → 信号真空的公司
- 对角线模式 → 各家公司在不同维度各有突破（健康）vs 集中在同一维度（同质化）

**模板 E：决策追踪表（每次巡田必出，放在报告末尾）**

纯 HTML 状态表。列=上次决策 | 状态（✅完成/⏳进行中/❌未执行/🟡降级）| 结果 | 本次动作。

关键规则：
- ❌ 连续出现 2 次 → 标记为 🔴 "需升级处理"
- ⏳ 超过 3 周 → 质疑是否还需要做
- 已完成但结果与预期不符 → 分析判断偏差

**模板 F：资源分配对比（非田消耗 > 田内消耗时）**

堆叠柱状图。每块田的实际消耗 vs 声明 budget。非田消耗单独一列。

### Widget 输出规则

1. **每个 widget 前一行写它回答的问题**（如"谁在变好谁在变差？"），让读者先知道要看什么
2. **每个 widget 后 ≤3 句话提取关键对比洞察**——不描述图表内容（图表自己会说话），只提取"所以呢？"
3. **至少 2 个 widget，最多 4 个**。2 个太少=信息密度不足，5 个太多=注意力分散
4. **颜色编码必须一致**：涨/好=Teal，跌/差=Coral，平/中性=Stone，新入=Sky，警告=Amber
5. **Widget 不重复文字分析**——图表展示 what，文字分析 why 和 so what
6. **加载 Chart.js CDN 一次**，多个 `<canvas>` 复用同一个库

### 对比洞察强制提取

每次巡田必须显式回答以下对比问题（可以嵌入 widget 或文字中）：

1. **最大赢家**：哪家公司本周修复进度变化最大？驱动因素是什么级别的信号？
2. **最大输家**：哪家公司恶化最严重？是周期性波动还是结构性恶化？
3. **最意外变化**：哪条信号与市场共识/此前预期最不一致？
4. **最安静公司**：哪家 ≥2 周无任何信号变化？是"没有消息就是好消息"还是"检测器没在工作"？
5. **组合健康度**：如果只看加权平均修复进度≈稳定，但内部方差是否在扩大？（表面平静 vs 暗流涌动）

## Design Notes

- 巡田和生活 change notes 的周 review 同一天做——两个 review 串起来
- 首次巡田（2026-07-06）建立了 5 田基线
- 巡田不替代每块田的独立 Agent 工作——巡田是元层的管理动作
- **Step 0 预检是 07-23 巡田 #6 的核心教训**——5 天检测器黑洞本可以通过预检在第一时间发现
- **WebSearch 补扫是 07-23 验证的降级路径**——质量不低于 Tavily，应注册为正式备用引擎
- **分派追踪是 07-19→07-21→07-23 三次巡田暴露的结构缺陷**——口头分配无追踪机制
- **休耕流程在 07-21 巡田 #5 首次执行**——Field 6 标记 fallow
- **可视化在 07-23 巡田 #6 首次试验**——3 个 widget 有效压缩了信息密度

## 巡田节奏

- **正常周**：Step 0→6 全流程，预计 ~15-20k token
- **检测器正常时**：Step 3c (WebSearch 补扫) 跳过，省 ~5k
- **轻量周**（无重大事件）：Step 3.5 (审计) 跳过，Step 6 可视化减为 1 个 widget
- **紧急周**（检测器断链 + 多事件）：全流程 + 全量可视化
