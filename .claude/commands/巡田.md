# /巡田 — Weekly Field Inspection

> 守望者的每周仪式。打开仪表盘，巡每一块田，做判断。

## Pipeline

### Step 1: 读仪表盘

Read `memory/FARM.md` — 了解上次巡田以来的变化。

### Step 2: 逐田巡查

对每块田（`memory/fields/*.yml`）：

1. 读 field YAML — 当前状态、上次收割、health notes
2. 检查：
   - 最后一次产出的日期 — staleness > 14d？
   - health 标记 — 有⚠️/❌？
   - harvest_trigger 条件是否满足？
3. 如果有对应 topic 文件，读 Quick Reference — 了解最近进展

### Step 3: 检查检测器信号

- 金融监管田：读 `~/.myagents/kb/d5_monitoring.db` 快照，最近一次 WoW Diff 有无变化
- 科技公司田：读 `memory/topics/cognitive-gap-watchlist.md`（追踪框架+公司骨架）和 `memory/fields/watchlist-scan-log.md`（最新扫描记录），7 家公司信号有无更新

### Step 3.5: 信号质量审计 ★

**当某家公司本周信号满足以下任一条件时，在决定"再种"（如启动 deep research）之前，必须先跑 cognitive-license 审计：**

触发条件：
- 修复进度变化 ≥ 5 个百分点
- 叙事突变标记触发
- 股价/市值出现异常波动（±10%以上）
- 新催化事件首次出现（产品发布/财报预告/融资公告）

审计流程：
1. 提取该公司的本周信号摘要（~200 字）
2. 跑 cognitive-license quick mode：`Workflow({ name: "cognitive-license", args: { text: signalSummary, mode: "quick" } })`
3. 判断：
   - FLAG = 0 → 信号可靠，可以基于它做决策
   - FLAG = 1-2 → 信号方向对但具体数字不可信。决策时只用方向，不用数字
   - FLAG ≥ 3 → 信号质量不足。只记录不行动。标注为"待人工核实"

**寒武纪案例教训**：组合修复-7%、股价-16.2%、"预告异常"标签——三条都是高强度信号，叠加产生"必须 deep research"的判断。但每条信号本身的信息质量从未被独立审计。如果 cognitive-license 发现"修复进度"是自定义未操作化指标、"16.2%"无数据源、"异常"是诠释标签——那些数字就不应该驱动一次 costly 的 deep research 决策。

### Step 4: 判留/弃/再种

对标记了 ready_for_harvest 或汤姆关注的产出：

- **留** — 有价值 → 更新对应 topic 文件，更新 field YAML last_harvest
- **弃** — 没价值 → 在 FARM.md 巡田发现中记原因
- **再种** — 方向对但不够 → 更新 field YAML 的 seed.key_claims

### Step 5: 更新仪表盘

更新 `memory/FARM.md`：
- 更新"上次巡田"日期
- 更新田块状态表
- 写本周巡田发现
- 更新资源消耗估算
- 更新需要决策的清单

### Step 5.5: 潜规则洞察

> 仪表盘告诉你每块田的显性状态。unwritten 镜头告诉你它们共同指向的系统性裂缝。

对农场状态跑 unwritten 分析——不是描述"各田健康分"，而是找跨田模式。

**裂缝** — 哪些故障在跨田复制？不列每块田的问题。找同一类问题出现在两块以上田的模式：检测器未接入同时出现在 Field 1 和 Field 2，是同一个根因吗？静默失败——Field 3 的 cron success 不等于 AI 产出，和 6/17 速递静默失败同源？告警疲劳——"major"标记存活超过 3 周，已经被注意到了但不再被处理。

**资源格局** — 实际分配 vs 声明分配。对比 field YAML 的 budget 声明和实际 token 消耗：谁在超支？谁在吃低保？priority: 1 的田实际 token 是否少于 priority: 4 的田？跨田授粉规则写了但从未被执行——是设计问题还是执行问题？

**仪表盘盲区** — 什么信号根本没被采集。FARM.md 不会告诉你：某块田的 Agent 连续 N 次产出被判"弃"——是田的问题还是判的问题？两块田在争同一个 API key 但没被建模。有对话中反复出现的需求但没建田。

**一句话诊断** — 不超过 30 字概括农场真实状态。不是"5 块田 3 块活跃 2 块待定"（这是仪表盘第一行已经说了的）。是"两块核心田空转三周，资源被按需田吸走，检测器缺陷是根因不是症状"。

输出方式：作为巡田报告的最后一个 section——`## 🕳️ 潜规则洞察`。和其他 section 并列，不替代它们。巡田先看显性状态，最后用 unwritten 镜头扫一遍盲区。

### Step 6: 输出巡田报告

**输出质量规则**（cognitive-license 审计后新增）：

1. **禁止裸标签** — "稳定""健康""正常"必须有可观测指标支撑。不能只说"Flash cron 稳定"——必须说"近7次执行成功率7/7，最近异常2026-07-10（Tavily超配额→自动切Exa已恢复）"
2. **所有数字标注来源** — 精确数字（百分比/价格/日期窗口）必须附带来源标记：`[实测]`/`[估算]`/`[引用:来源]`。42%组合修复必须标注是实测还是估算
3. **预测必须用非确定语气** — 未来事件标注"据称""市场预期""截至X日"。不能写"明天发布"，要写"据商汤官方公告，预计7/18 WAIC发布"
4. **效率数字只报行数，不推断认知** — "阅读量减少73%"→改为"骨架1290行，原1785行"。不要把行数算术等同于认知效率提升

```
🌾 巡田报告 — YYYY-MM-DD

本周收割：N 块田
  - [田名]：[留/弃/再种] — [一句话原因]

  ✅ N 块健康
  ⚠️ N 块需关注 — [具体问题]
  ❌ N 块异常 — [具体问题]

需要决策：
  - [决策项 1]
  - [决策项 2]

下次巡田：YYYY-MM-DD
```

## Design Notes

- 巡田和生活 change notes 的周 review 同一天做——两个 review 串起来
- 首次巡田（2026-07-06）建立了 5 田基线
- 巡田不替代每块田的独立 Agent 工作——巡田是元层的管理动作
